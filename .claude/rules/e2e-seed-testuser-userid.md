# E2E Seeds Must Use `testUser.userId`, Not the `user-1` Default

In dev_bypass E2E tests, the per-test identity is `testUser.userId` (generated by `buildE2EUserId(testInfo)`), **not** `user-1`. The `tw_e2e_user` cookie holds this ID; `apps/web/lib/api.ts` forwards it as `x-user-id`; the API's `resolveDevBypassFallback` consumes it.

The constant `"user-1"` only applies when no `x-user-id` header is sent. Helpers that seed state via direct HTTP with `x-user-id: "user-1"` produce state owned by user-1 — not the current test user. Any UI assertion that reads the test user's owner-scoped view of that state will fail.

**Symptoms:**
- Seeded state exists in the DB (verifiable via a separate HTTP check) but the UI shows an empty list.
- Arrange looks correct; the first UI assertion times out waiting for a row keyed by a seeded id.

**Fix:** destructure `testUser` from the fixture and pass `testUser.userId` to seed helpers when the spec needs the UI to observe what was seeded:

```ts
test("...", async ({ sharing, testUser }) => {
  const grantee = await seedUser({ ... });
  const { shareId } = await seedResolvedShareFromAdmin(grantee.email, testUser.userId);
  //                                                                 ^^^^^^^^^^^^^^^^^
  await sharing.actions.navigateToSharing();
  await sharing.assert.outboundRowVisibleWithEmail(shareId, grantee.email);
});
```

For seeds whose visibility is grantee-scoped (inbound views) or admin-scoped (audit logs), the owner identity is irrelevant — the `user-1` default is fine.

**Why:** Discovered in KZO-145 scenario 15 (revoke + notification). The Playwright cookie-jar fix (see `playwright-oauth-cookie-patterns.md`) cleared a 403 and uncovered this owner-mismatch bug. The seeded share was owned by `user-1` but the UI rendered as `e2eUserId`'s portfolio, so the outbound row never appeared. KZO-146 followed the pattern correctly from the start — all switcher UI specs pass `owner.userId` explicitly.

**How to apply:** When writing an E2E spec that (a) seeds state via HTTP and (b) asserts through the UI that the seeded state is visible from the test user's perspective, pass `testUser.userId` as the owner/actor to the seed helper. Default `"user-1"` only for specs where the seeded state's ownership is irrelevant to the assertion path.
