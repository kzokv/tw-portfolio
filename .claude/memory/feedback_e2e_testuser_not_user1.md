---
name: E2E test user is not "user-1" in dev_bypass
description: The default E2E test user has a per-test e2eUserId, not "user-1" — admin-scoped HTTP seeds must use testUser.userId when the spec observes/mutates through the UI
type: feedback
---

In dev_bypass E2E tests, the default test user has a per-test `e2eUserId` (generated by `buildE2EUserId(testInfo)`), **not** `user-1`. The `tw_e2e_user` cookie is set to this `e2eUserId` in `assignIdentity`; `lib/api.ts` forwards it as `x-user-id`; the API's `resolveDevBypassFallback` uses that ID.

The constant `"user-1"` only applies when **no** `x-user-id` header is sent (fallback default). Helpers that seed state via direct HTTP with `x-user-id: "user-1"` produce state owned by user-1 — not the current test user. Any UI assertion that relies on the test user's outbound/owner-scoped views of that state will fail.

**Symptoms:**
- Seeded state exists in DB (verified via separate HTTP check) but the UI shows an empty list.
- The spec's Arrange looks right but the first UI assertion times out waiting for a row keyed by a seeded id.

**Fix:** destructure `testUser` from the fixture and pass `testUser.userId` to seed helpers when the spec needs the UI to observe what was seeded:

```ts
test("...", async ({ sharing, testUser }) => {
  const grantee = await seedUser({ ... });
  const { shareId } = await seedResolvedShareFromAdmin(grantee.email, testUser.userId);
  //                                                                 ^^^^^^^^^^^^^^^^^
  await sharing.actions.navigateToSharing();
  await sharing.assert.outboundRowVisibleWithEmail(shareId, grantee.email);
});
```

For seeds whose visibility is grantee-scoped (inbound views) or admin-scoped (audit logs), the owner identity doesn't matter — `"user-1"` default is fine.

**Why:** Discovered in KZO-145 E2E scenario 15 (revoke + notification) — the cookie-jar fix (see `feedback_playwright_request_cookie_jar_leak.md`) cleared the 403 but uncovered this owner-mismatch bug. The seeded share was owned by `user-1` but the UI rendered as `e2eUserId`'s portfolio, so the outbound row never appeared.

**How to apply:** When writing an E2E spec that (a) seeds state via HTTP and (b) asserts through the UI that the seeded state is visible from the test user's perspective, pass `testUser.userId` as the owner/actor to the seed helper. Default `"user-1"` only for specs where the seeded state's ownership is irrelevant to the assertion path.
