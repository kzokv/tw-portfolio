# Cash Ledger Entry Rules

Reference document for `cash_ledger_entries` — authoritative source for how entries are generated, constrained, linked, and corrected. For entity definitions and lifecycle overview, see [canonical-accounting-model.md](./canonical-accounting-model.md).

---

## Entry Type Table

| Type | Enforced Sign | Generated By | Required Linkage |
|---|---|---|---|
| `TRADE_SETTLEMENT_IN` | `amount > 0` | SELL trade booking; `replayPositionHistory` | `relatedTradeEventId` |
| `TRADE_SETTLEMENT_OUT` | `amount < 0` | BUY trade booking; `replayPositionHistory` | `relatedTradeEventId` |
| `DIVIDEND_RECEIPT` | `amount > 0` | `postDividend`; `preparePostedCashDividendUpdate` | `relatedDividendLedgerEntryId` |
| `DIVIDEND_DEDUCTION` | `amount < 0` | `postDividend`; `preparePostedCashDividendUpdate` | `relatedDividendLedgerEntryId` |
| `MANUAL_ADJUSTMENT` | any sign | reserved — no active code path | none required |
| `REVERSAL` | any sign | reserved — no active code path | `reversalOfCashLedgerEntryId` |

`DEPOSIT` and `WITHDRAWAL` are planned future first-class types. Not yet in schema; deferred per KZO-27 scope-grill Q6.

## Sign Conventions

Sign per type is enforced by a DB `CHECK` constraint (`baseline_current_schema.sql:359-364`):

```sql
CHECK (
  (entry_type = 'TRADE_SETTLEMENT_IN'  AND amount > 0)
  OR (entry_type = 'TRADE_SETTLEMENT_OUT' AND amount < 0)
  OR (entry_type = 'DIVIDEND_RECEIPT'  AND amount > 0)
  OR (entry_type = 'DIVIDEND_DEDUCTION' AND amount < 0)
  OR entry_type IN ('MANUAL_ADJUSTMENT', 'REVERSAL')
)
```

Additional constraints:
- `CHECK (amount <> 0)` — zero-amount entries are invalid. Replay filters before insert (invariant 4 of `replayPositionHistory`).
- `NUMERIC(20, 2)` — supports fractional ETF settlement amounts.

**MemoryPersistence gap:** Sign conventions and the zero-amount guard are DB-enforced only. `PERSISTENCE_BACKEND=memory` (dev/test) silently accepts sign violations and zero-amount entries. Do not rely on sign correctness in memory-mode tests.

## Entry Generation Matrix

### Trade Settlement Paths

1. **`createTransaction` (`portfolio.ts:120`)** — calls `buildTradeSettlementCashEntry` at booking time; one entry per trade. ID pattern: `cash-{tradeId}`. Source: `"trade_settlement"`.
2. **`replayPositionHistory` (`replayPositionHistory.ts:126-149`)** — reconstructs settlement entries from the full trade set during replay. ID pattern: `cash-replay-{tradeId}`. Source: `"trade_settlement"`.
3. **`confirmRecompute` (`recompute.ts:84`)** — calls `replaceCashLedgerEntryForTrade` when fees change on an existing trade. Reuses the `cash-{tradeId}` ID.

Settlement amount formula:
- BUY: `-(quantity × unitPrice + commission + tax)`
- SELL: `quantity × unitPrice - commission - tax`

### Dividend Cash Paths

1. **`postDividend` (`dividends.ts:222`)** — calls `buildDividendCashLedgerEntries`; produces one `DIVIDEND_RECEIPT` + N `DIVIDEND_DEDUCTION` entries.
2. **`preparePostedCashDividendUpdate` (`dividends.ts:319`)** — same builder; entries replaced via delete-and-replace when a posted dividend is updated.

ID patterns:
- Receipt: `{dividendLedgerEntryId}:receipt`
- Deduction: `{dividendLedgerEntryId}:deduction:{deductionId}`

Deduction subtypes that produce `DIVIDEND_DEDUCTION` entries (type `DividendDeductionType` in `store.ts:125`):
`NHI_SUPPLEMENTAL_PREMIUM`, `WITHHOLDING_TAX`, `BROKER_FEE`, `BANK_FEE`, `TRANSFER_FEE`, `CASH_IN_LIEU_ADJUSTMENT`, `ROUNDING_ADJUSTMENT`, `OTHER`.

**Zero-amount deduction guard:** `buildDividendCashLedgerEntries` produces `amount: -deduction.amount`. A zero-amount deduction yields a zero-amount cash entry, violating `CHECK (amount <> 0)`. The client-side form filters zero-amount deductions before submission; `dividend_deduction_entries` enforces `CHECK (amount > 0)` at the DB level in Postgres. MemoryPersistence has neither guard. Filtering zero-amount deductions in the service layer is a separate tracked bug.

### ID Collision Prevention

Trade and dividend cash entries share the same table but use distinct `source` values — `"trade_settlement"` vs `"dividend_posting"`. The unique partial index `ux_cash_ledger_entries_account_source_reference` on `(account_id, source, source_reference) WHERE source_reference IS NOT NULL` prevents collisions within each namespace.

## Linkage Rules

FK linkage is enforced by a DB `CHECK` constraint (`baseline_current_schema.sql:366-374`):

| Entry type | `relatedTradeEventId` | `relatedDividendLedgerEntryId` |
|---|---|---|
| `TRADE_SETTLEMENT_IN` / `OUT` | required (NOT NULL) | must be NULL |
| `DIVIDEND_RECEIPT` / `DEDUCTION` | must be NULL | required (NOT NULL) |
| `MANUAL_ADJUSTMENT` / `REVERSAL` | optional | optional |

**Cascade behavior:**
- `related_trade_event_id` references `trade_events(id) ON DELETE CASCADE` — deleting a trade event hard-deletes its linked cash entries.
- `related_dividend_ledger_entry_id` has no `ON DELETE CASCADE` — dividend cash entries are managed explicitly by the dividend posting service, not by FK cascade.

**`receivedCashAmount` derivation:** Not stored on `dividend_ledger_entries` (column removed in migration 010). Computed on every read as:

```sql
COALESCE(
  SUM(amount) FILTER (WHERE entry_type = 'DIVIDEND_RECEIPT')
  FROM cash_ledger_entries
  WHERE related_dividend_ledger_entry_id = dle.id,
  0
)
```

Consequence: deleting or editing a `DIVIDEND_RECEIPT` entry retroactively changes the reported `receivedCashAmount`.

## Correction Model

Cash ledger entries are **mutable derived state**. The source of truth is the originating event — trade event or dividend ledger entry — not the cash entry itself.

Sanctioned correction patterns:

1. **Trade settlements — full reconstruction:** `replayPositionHistory` deletes all `TRADE_SETTLEMENT_IN/OUT` for the scoped `(account_id, ticker)` and rebuilds from the current trade set. This is a full reconstruction, not a point correction.

2. **Trade settlements — fee-only update:** `replaceCashLedgerEntryForTrade` (called by `confirmRecompute`) replaces a single trade's cash entry in the in-memory store when only fees change.

3. **Dividend cash entries — delete-and-replace:** `updatePostedCashDividend` in persistence deletes all cash entries linked to the dividend ledger entry and re-inserts from the updated posting.

The `REVERSAL` type remains in the schema as a reserved capability. It has no active code path in Phase 1. Direct edit and delete-and-replace are the sanctioned correction patterns.

## Replay Reconstruction Semantics

`replayPositionHistory` (`apps/api/src/services/replayPositionHistory.ts`) interacts with cash entries in steps 4–6:

- **Step 4 — Delete:** `deleteTradeCashEntriesForAccountTicker(userId, accountId, ticker)` removes all `TRADE_SETTLEMENT_IN/OUT` for the scoped `(account_id, ticker)`. This is a targeted delete, not a full-user wipe.
- **Step 5 — Rebuild:** The replay loop processes trades in `trade_date ASC, booking_sequence ASC` order and generates one settlement entry per trade. Zero-amount entries are filtered before accumulation (`replayPositionHistory.ts:133-134`).
- **Step 6 — Persist:** `bulkInsertCashLedgerEntries(userId, allCashEntries)` writes the rebuilt set.

Dividend cash entries (`DIVIDEND_RECEIPT`, `DIVIDEND_DEDUCTION`) are **not touched by replay**. They are managed independently by the dividend posting service and remain unchanged when position history is replayed.
