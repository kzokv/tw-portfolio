## HIGH

### H1 — `getFxRate.integration.test.ts:214-222` — Nested `it()` makes NUMERIC coercion test a silent no-op

```ts
// Lines 214-222 as written:
it("result is a JS number (not a string) — pg NUMERIC coercion (D12)", () => {
  // ... comment ...
  it("getFxRate returns a JS number, not a string", async () => {   // ← nested it()
    await seedRate("2026-04-24", "USD", "TWD", 31.5);
    const result = await persistence!.getFxRate("USD", "TWD", "2026-04-24");
    expect(typeof result).toBe("number");
  });
});
```

In Vitest, a bare `it()` call inside a running `it()` callback does **not** register a new test — the inner `it` is silently discarded. The outer `it` has **no assertions** and passes vacuously. The intended assertion (`typeof result === "number"`) is never executed.

This matters because `pg` returns `NUMERIC` columns as strings. `getFxRate` uses `Number(result.rows[0].rate)` which is correct, but the TypeScript return type (`Promise<number | null>`) and the `Number(...)` cast are the only guards — there is no runtime test verifying the coercion path.

**Recommendation:** Unwrap the inner `it()` into a standalone top-level `it()` in the same `describePostgres` block:

```ts
it("result is a JS number, not a string — pg NUMERIC coercion (D12)", async () => {
  await seedRate("2026-04-24", "USD", "TWD", 31.5);
  const result = await persistence!.getFxRate("USD", "TWD", "2026-04-24");
  expect(typeof result).toBe("number");
  expect(result).toBeCloseTo(31.5, 6);
});
```

## LOW

### L1 — `currencyWalletAccounting.ts:30-34` — JSDoc comment contradicts the actual rounding behavior

The module-level JSDoc says:

```
 *   - Math inside this module uses full precision; rounding happens in the
 *     caller (currencyWalletSnapshotGeneration.ts) at snapshot-write time.
```

But `applyEntryToWalletState` itself calls `roundToDecimal(balance, 2)` and `roundToDecimal(pnl, 2)` in Cases A, B, and C. The code matches the architect-design pseudocode and is correct. The JSDoc is inaccurate — the sentence about "rounding happens in the caller" refers only to WAC's 8dp rounding (done in `buildSnapshot`), not to balance/PnL which are rounded at each step inside the module.

**Recommendation:** Replace that sentence with:

```
 *   - balance and realizedFxPnlLifetime: rounded to 2dp at each step within
 *     this module (consistent with architect-design §3 pseudocode).
 *   - wacFxToUsd: full precision here; caller rounds to 8dp at snapshot-write.
```

No code change needed — the code is correct.

## INFORMATIONAL

### I1 — `currency-wallet-wac.integration.test.ts:295-305` — REVERSAL case 3 asserts conditionally; silent-pass risk when 0 rows emitted

```ts
if (result.totalRows > 0) {
  const rows = await readSnapshots(userId);
  for (const row of rows) {
    expect(row.wac_fx_to_usd).toBeNull();
    expect(Number(row.realized_fx_pnl_lifetime)).toBe(0);
  }
}
// zero rows → test passes with no assertion
```

If the REVERSAL filter was broken and produced zero rows (e.g. via a logic error that silently dropped both entries), the test would still pass. Per D7, emitting zero rows is semantically correct (net balance = 0 → no snapshot). But the test's D7 invariant — that the WAC walker was not influenced by the pair — is only exercised if rows ARE emitted.

Optional hardening (not blocking): add an assertion that the REVERSAL entry itself was correctly persisted (via raw SQL count) and that the WAC state on any emitted rows reflects the filter worked. This is a nice-to-have, not a blocker.

`MissingFxRateError` is exported from `currencyWalletAccounting.ts` and has a well-defined interface, but no unit test verifies its constructor or `details` payload. This is acceptable per scope — the wallet generator doesn't throw it in KZO-166 (only write-path callers like a future FX-lookup route would), and the architect-design §5a doesn't require it. Flag for KZO-168 test coverage.
