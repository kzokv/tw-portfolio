## Findings

### HIGH — H1: `buildCreateAccountHref()` generates dead drawer URL (user-facing breakage)

**File:** `apps/web/components/portfolio/AddTransactionCard.tsx:120-131`

```ts
// Lines 127-131 — BROKEN:
params.set("drawer", "settings");    // drawer is deleted
params.set("settingsTab", "accounts");
params.set("accountsPrefillCurrency", currency);
return `${pathname}?${params.toString()}`;
```

The `buildCreateAccountHref()` function still generates `?drawer=settings&settingsTab=accounts&accountsPrefillCurrency={currency}`. With the drawer deleted, clicking this link (shown to users on the transaction form when they have no account for the selected currency) navigates to the DASHBOARD with no settings UI. The link becomes a dead end.

**Compounding gap:** `AccountsSettingsClient.tsx:191-194` mounts `<AccountCreateForm>` without a `prefillCurrency` prop and without reading `accountsPrefillCurrency` from URL search params. Even if the URL were updated to `/settings/accounts?accountsPrefillCurrency=USD`, the prefill would not work because the component doesn't read it.

**User impact:** KZO-169 NC4 feature (deep-link to account creation with prefill currency) is broken.

**Fix:**
1. Update `buildCreateAccountHref()` to produce `/settings/accounts?accountsPrefillCurrency={currency}` (use `encodeURIComponent`)
2. In `AccountsSettingsClient`, add `useSearchParams()` and pass `prefillCurrency` to `AccountCreateForm`:
   ```ts
   const searchParams = useSearchParams();
   const prefillCurrency = searchParams.get("accountsPrefillCurrency") as AccountDefaultCurrency | null;
   // ...
   <AccountCreateForm prefillCurrency={prefillCurrency ?? undefined} .../>
   ```

**Companion:** Also update unit test `AddTransactionCard.test.tsx:103,106` to assert the new `/settings/accounts?...` URL format (see M2).

### LOW — L1: Null guard dead code in `useAutoSave` and `useConfirmedSave`

**Files:**
- `apps/web/features/settings/hooks/useAutoSave.ts:115,132`
- `apps/web/features/settings/hooks/useConfirmedSave.ts:86`

```ts
// useAutoSave.ts:115 — guard never fires for null:
if (pending === null && typeof pending !== "object") {
  // typeof null === "object" in JavaScript
  // → condition is: null === null && "object" !== "object" → null && false → never true
```

Same pattern at useAutoSave.ts:132 and useConfirmedSave.ts:86.

**Practical impact: LOW.** JavaScript is single-threaded; the race condition the guard is protecting against cannot occur in synchronous execution. Current `TValue` usages (string fields) are never null. The guard was intended to be `if (pending === null) return;`.

**Fix:** Replace with:
```ts
if (pending === null) return;
```
