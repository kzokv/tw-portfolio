### MEDIUM-3 — `app/sharing/page.tsx` missing `initialSidebarOpen` (item 14 partial miss)

**File:** `apps/web/app/sharing/page.tsx` + `apps/web/app/sharing/layout.tsx`  
**Rule:** 17-item preservation checklist item 14; SSR sidebar-cookie pipeline (architect-design §4)

All page routes except `/sharing` pass `initialSidebarOpen` sourced from `readSidebarStateCookie()`. The sharing page uses a `"use client"` wrapper (`SharingRouteProvider`) with its own context. Neither `app/sharing/layout.tsx` (which is a server component) nor the page itself call `readSidebarStateCookie()` or thread `initialSidebarOpen` into AppShell.

Result: Users who have collapsed their sidebar will see it re-expand when navigating to the sharing page. On desktop this is a visible FOUC; the sidebar snaps from open → closed on first client paint. This is a real UX regression for users with a saved collapsed preference.

**Fix (two options, pick one):**

Option A — Add `initialSidebarOpen` in `app/sharing/layout.tsx`:
```ts
// apps/web/app/sharing/layout.tsx
import { readSidebarStateCookie } from "../../lib/sidebar-cookie";
export default async function SharingLayout({ children }: { children: React.ReactNode }) {
  const sidebarOpen = await readSidebarStateCookie();
  return <SharingRouteProvider initialSidebarOpen={sidebarOpen}>{children}</SharingRouteProvider>;
}
```
Then `SharingRouteProvider` and `SharingPage` thread `initialSidebarOpen` through to `AppShell`.

Option B — Call `readSidebarStateCookie()` directly in the server layout and inject it as a Next.js `searchParam`-style mechanism. This is more complex; Option A is preferred.

### LOW-1 — `avatar-menu-admin` alias is on the wrapper `<div>`, not on `<a>` (design intent mismatch)

**File:** `apps/web/components/profile/ProfileMenu.tsx` lines 233–240  
**Rule:** 17-item preservation checklist item 12; architect-design §7 ("the wrapper div carries the legacy `avatar-menu-admin` testid so existing OAuth specs that `getByTestId('avatar-menu-admin').click()` still hit a clickable element")

Current structure:
```tsx
<div data-testid="avatar-menu-admin">
  <DropdownMenuItem asChild>
    <a href="/admin" data-testid="profile-menu-admin-link">
```

The design comment in the spec says "existing OAuth specs that `getByTestId('avatar-menu-admin').click()` still hit a clickable element (the descendant `<a>`)." This is actually the correct interpretation — the div carries the legacy testid, the `<a>` carries the new canonical testid. A `.click()` on the div will bubble to the link child in Playwright. The functional behavior is preserved.

However, the architect-design §7 also says "The Admin entry carries both `data-testid='profile-menu-admin-link'` and the legacy `data-testid='avatar-menu-admin'`" (emphasis: carries BOTH on the same element). The spec's comment in the implementation (`ProfileMenu.tsx:231-233`) also references "the wrapper div carries the legacy testid so... `getByTestId('avatar-menu-admin').click()` still hit a clickable element (the descendant `<a>`)."

The current implementation matches the inline code comment's intent and works functionally. If any OAuth spec uses `getByTestId("avatar-menu-admin").click()` expecting the click to navigate (not just locate a div), the bubble will work. **No functional fix required** — but the Architect should confirm the inline comment is the authoritative intent and update the spec wording to avoid future reviewer confusion.

**Recommendation:** No code change. Architect to update architect-design §7 wording to clarify that the alias is intentionally on the wrapper div (not the same element as the canonical testid).
