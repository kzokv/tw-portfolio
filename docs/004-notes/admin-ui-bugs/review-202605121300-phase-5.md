### MEDIUM (must fix before merge)

#### MEDIUM-1 — Indentation inconsistency in moved `<Card>` blocks (`AdminSettingsClient.tsx:691-831` and `837-864`)

**File:** `apps/web/components/admin/AdminSettingsClient.tsx`  
**Lines:** 691–832 (`display-defaults` Card), 837–864 (`api-keys` Card)

When the two orphan `<Card>` blocks were moved inside their respective `<TabsContent>` wrappers, the outer `<Card>` tag was re-indented to 10-space (correct for the new nesting depth inside `<TabsContent>`), but the inner `<div className="space-y-5">` and all its children retained the original 8-space indentation. The result is that `</Card>` closes at 10-space indent while its direct child `<div>` opens at 8-space — two spaces shallower than the opening `<Card>`.

```tsx
// Current (incorrect) — Card at 10-space, child div at 8-space
        <TabsContent value="display-defaults" data-testid="admin-settings-panel-display-defaults">
          {/* ── KZO-159: Dashboard Timeframe Defaults section ─────────────── */}
          <Card data-testid="timeframe-defaults-section">    ← 10-space (correct)
        <div className="space-y-5">                          ← 8-space (wrong — should be 12-space)
          <div>
            ...
          </div>
          </Card>                                            ← 10-space (correct)
```

This is cosmetically inconsistent and signals the move was done with minimal indentation adjustment. It does not break the render, but it violates the surrounding file's consistent 2-space-per-level convention and will cause formatter-triggered diff noise in follow-up PRs.

**Fix:** Re-indent the `<div className="space-y-5">` block and all its children inside both Card blocks by +2 spaces (from 8-space to 12-space for the direct child). Same applies to the closing `</div>` and all intermediate elements.

**Both affected regions:**
- `AdminSettingsClient.tsx:692` — `<div className="space-y-5">` (display-defaults Card body)
- `AdminSettingsClient.tsx:838` — `<div className="space-y-5">` (api-keys Card body)

### INFORMATIONAL

**I-1 — Popover dismiss test uses `!(await content.isVisible())` without `waitFor` (potential flakiness risk)**  
`provider-health-aaa.spec.ts:498` (popover-B outside-click) and `:519` (popover-C Escape) check dismissal with `!(await content.isVisible())`. Radix Popover closes synchronously via `onInteractOutside`, so this is unlikely to race in practice, but it deviates from the project's usual `waitFor({ state: "hidden" })` pattern. Low risk given Radix's synchronous dismiss model; flagged for awareness if flakes appear post-merge.

**I-2 — KZO-199 `truncate` on `CardDetail` props is unchanged (not in scope, but context)**  
`AdminProvidersClient.tsx:389–393` — `<CardDetail ... truncate />` props are still passed for last-success, last-failed, error counts, and rate-limit counts in the card variant. Per the scope-todo, only the provider-name cell's `truncate` was in scope for removal; the CardDetail truncation for other fields is intentional (numeric/timestamp card details that should clip, not wrap). This is correct and not a finding — recorded here for the KZO-206 rule-authoring step to document the residual truncate scope.

**I-3 — `ensureDisplayDefaultsTabActive` in AppShellAssert uses `this.uiActions.click.perform` for the tab click**  
The private helper in `AppShellAssert.ts` calls `await this.uiActions.click.perform(trigger)` to activate the display-defaults tab if not already active. The pattern is consistent with existing `ensureAdminSettingsTabActive` in `AppShellActions.ts`. No action required.

**I-4 — `navigateToAdminSettingsTab` in AppShellActions is a pure alias for `ensureAdminSettingsTabActive`**  
The new `navigateToAdminSettingsTab` delegates entirely to `ensureAdminSettingsTabActive`. The dual-name situation is intentional per architect-design.md ("preferred call site name"). The union type on the `slug` parameter is duplicated across both methods. This is acceptable cosmetically; a future refactor could extract the union to a named type. Not a blocking concern.
