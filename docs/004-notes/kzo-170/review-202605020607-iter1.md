## CRITICAL

### CRITICAL-1 — `MarketDataEndpoint.getPrice` sends `marketCode` (camelCase) but the route requires `market_code` (snake_case)

- **File:** `libs/test-api/src/endpoints/MarketDataEndpoint.ts:27`
- **Bug:** the endpoint serializes `params.set("marketCode", marketCode)`. The route at `apps/api/src/routes/registerRoutes.ts:3048` parses the query with Zod `{ market_code: z.enum([...]) }`. Camel- and snake-case keys are different objects to `URLSearchParams` + Zod — no mapping happens.
- **Blast radius:**
  - `market-data-price-aaa.http.spec.ts` M1 (TW happy path): expects 200 + close=1005, will get 400.
  - M2 (US dispatch): expects 200, will get 400.
  - M3 (missing marketCode → 400): passes for the wrong reason (the param is *never* sent in any test case).
  - M4 (invalid value → 400): passes for the wrong reason (still: missing param).
- **Convention pin:** `apps/web/features/portfolio/services/portfolioService.ts:117` correctly sends `market_code: marketCode`. `apps/api/test/http/specs/transaction-form-market-code-aaa.http.spec.ts` (KZO-169) also uses `market_code=...` snake-case query in `/instruments?market_code=TW` etc. The route is the source of truth; the test endpoint diverges.
- **Recommendation:** change `params.set("marketCode", marketCode)` to `params.set("market_code", marketCode)` in `MarketDataEndpoint.ts:27`. No other file changes — `MarketDataApiActions.getPrice` and the spec keep their `marketCode` arg names (those are typed JS, internal to the test wrapper).

### CRITICAL-2 — `registerCatalogSyncWorker` skips daily-refresh on partial completion + reschedule, contradicting D12 + the QA test

- **File:** `apps/api/src/services/market-data/registerCatalogSyncWorker.ts:140`
- **Bug:** `if (!rescheduled && completedMarkets.length > 0) { await enqueueDailyRefreshFn(...); }`. When TW completes and US throws `RateLimitedError`, `rescheduled=true` short-circuits the enqueue.
- **Locked design** (scope-todo D12): *"Rate-limited markets reschedule with `singletonKey: CATALOG_SYNC_QUEUE`; daily-refresh enqueue runs for completed markets immediately."*
- **QA test** (`apps/api/test/unit/catalogSyncReschedule.test.ts:117`):
  ```ts
  // Daily refresh STILL fires for the completed (TW) markets — completed work
  // shouldn't wait on the rate-limited market to recover.
  expect(enqueueDailyRefreshFn).toHaveBeenCalledTimes(1);
  ```
  This assertion fails against the current implementation (call count 0).
- **Implementation comment is also internally contradictory:** lines 132–137 say *"partial completion still wants to enqueue daily-refresh for the markets that DID finish — those tickers won't get re-touched until tomorrow's cron otherwise"*; line 138 then says *"We skip … when we rescheduled (the rescheduled run will own the eventual daily-refresh enqueue)."* Pick one. The locked design + test pick the former.
- **Recommendation:** drop the `!rescheduled` guard:
  ```ts
  if (completedMarkets.length > 0) {
    await enqueueDailyRefreshFn(boss, persistence, log);
  }
  ```
  This still skips the no-op case (`completedMarkets === []`) and matches both the test and the comment-block intent.

### MEDIUM-3 — `MockFinMindUsStockMarketDataProvider`'s `fixtureStartDate` constructor variant is implemented but not exercised by any test in this PR

  - `apps/api/src/services/market-data/providers/mockFinmindUsStock.ts:114-118` — accepts `opts?: { fixtureStartDate?: string }`.

Could be deferred to KZO-186 splits work which will likely add more US fixture-shape tests.
- **Routing:** Phase 4 (Implementer can add the test if scope allows) OR a follow-up ticket. Mark as advisory.
