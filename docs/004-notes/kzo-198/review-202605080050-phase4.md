## HIGH

### H1 — Tier 2 fields included in `patchAdminSettingsSchema` — violates scope-todo "SQL only"

**File:** `apps/api/src/routes/adminRoutes.ts` — `patchAdminSettingsSchema` (z.object body, approx. line 134–160) and `TIER1_PLAIN_FIELDS` array (approx. line 162–182).

**What was found:**

`patchAdminSettingsSchema` accepts 17 plain fields + 2 Tier 0 = 19 total. This includes 5 Tier 2 fields:
- `dailyRefreshLookbackDays`
- `dailyRefreshPriority`
- `sseHeartbeatIntervalMs`
- `sseMaxConnectionsPerUser`
- `sseBufferDefaultTtlMs`

The `TIER1_PLAIN_FIELDS` constant (internal to the PATCH handler) lists all 17, including the 5 above.

**Design says:**

`design.md §4`:
> "extend `patchAdminSettingsSchema` with all **12 Tier 1 + 2 Tier 0** fields"

`scope-todo (locked):` Tier 2 = "Plain DB-only escape hatch | UI: **none (SQL only)**"

12 Tier 1 = 6 rate limits + 3 provider health + 3 backfill (retry limit, retry delay, finmind 402 retry ms).

**Impact:**

The admin PATCH endpoint now accepts Tier 2 fields, bypassing the "SQL only" restriction. While no UI form renders for them (AdminSettingsClient correctly has no Tier 2 form controls), any API client or curl command can write Tier 2 fields via `PATCH /admin/settings`. This also stamps `value_change` audit rows for Tier 2 writes made through the API, while SQL writes (the intended path) produce no audit rows — creating inconsistent auditability.

**Fix:**

1. Remove the 5 Tier 2 fields from `patchAdminSettingsSchema`.
2. Remove them from `TIER1_PLAIN_FIELDS` (or rename to `TIER1_OVERRIDE_FIELDS` to match the reduced set).
3. If informational display of Tier 2 effective values in GET `/admin/settings` is desired (read-only), keep them in the DTO — that's acceptable. The violation is PATCH accepting them, not GET returning them.

**Note on DTO:** The `AppConfigDto` in shared-types marks these as "no UI form, surfaced for completeness" — read-only display is fine. The DTO comment and `loadAppConfigDto` need no change for the GET path.

## MEDIUM

### M1 — `repairCooldownMinutes` UI bounds hardcoded; does not consume DTO bounds

**File:** `apps/web/components/admin/AdminSettingsClient.tsx:24–25, 354–355, 367`

**What was found:**

```ts
const MIN_MINUTES = 1;        // line 24
const MAX_MINUTES = 10080;    // line 25
```

These are used in the repair-cooldown section (HTML `min`/`max` attrs, validation message). All new KZO-198 fields correctly use `config.bounds.fieldName.min/max` from the DTO, but the pre-existing `repairCooldownMinutes` section was not updated.

**Design says:**

`scope-todo (locked):` "Bounds defined once in `apps/api/src/services/appConfig/bounds.ts`, exported and consumed by both API schema and **UI form** `min`/`max`."

`bounds.ts` already exports `APP_CONFIG_BOUNDS.repairCooldownMinutes = { min: 1, max: 10080 }` and `loadAppConfigDto` includes `bounds: APP_CONFIG_BOUNDS` in the DTO — so the single-source machinery is in place.

**Fix:**

Replace `MIN_MINUTES` / `MAX_MINUTES` with `config.bounds.repairCooldownMinutes.min` / `config.bounds.repairCooldownMinutes.max` at each usage site (validation function line 64–65, `min=`/`max=` attrs lines 354–355, helper text line 367). The pre-existing `repairCooldownMinutes` form should be refactored to use `NumericOverrideRow` with a `bounds` prop — same pattern as the new KZO-198 fields (lines 631–681).

### M2 — `MaskedSecretInput` hardcodes Tier 0 length bounds; does not accept DTO `secretLengthBounds`

**File:** `apps/web/components/admin/MaskedSecretInput.tsx:19–20`

**What was found:**

```ts
const MIN_SECRET_LENGTH = 20;
const MAX_SECRET_LENGTH = 500;
```

These are hardcoded module-level constants. The DTO carries `secretLengthBounds: { min: 20, max: 500 }` (from `APP_CONFIG_SECRET_LENGTH` in bounds.ts), and `AdminSettingsClient.tsx` has the DTO in scope, but never passes `secretLengthBounds` as a prop to `MaskedSecretInput`.

**Fix:**

Add `secretLengthBounds?: { min: number; max: number }` to `MaskedSecretInputProps`. Default to `{ min: 20, max: 500 }` when not provided (backwards-compatible). Pass `config.secretLengthBounds` from AdminSettingsClient at the two `MaskedSecretInput` call sites (lines 781–800). Update the validation function to use `secretLengthBounds.min/max` instead of the constants.

### M3 — Migration COMMENT tier labels: 6 columns mislabeled (pre-merge correction window)

**File:** `db/migrations/047_kzo198_app_config_tier_a_constants.sql:69, 74, 76, 80, 85, 87`

**What was found:**

Six `COMMENT ON COLUMN` lines have incorrect tier labels relative to what the implementation (form controls, PATCH schema, DTO comments) actually treats them as:

| Column | COMMENT says | Implementation treats as | Expected label |
|---|---|---|---|
| `provider_error_trail_retention_days` | Tier 2 | Tier 1 (form control, in PATCH) | Tier 1 |
| `backfill_retry_limit` | Tier 2 | Tier 1 (form control, in PATCH) | Tier 1 |
| `backfill_retry_delay_seconds` | Tier 2 | Tier 1 (form control, in PATCH) | Tier 1 |
| `daily_refresh_lookback_days` | Tier 1 | Tier 2 (no form, in PATCH via H1) | Tier 2 |

Per `migration-strategy.md`: "Once a migration has been applied to any environment (dev, staging, prod), it is immutable." The pre-merge window is the ONLY time to correct these. Post-merge, a new migration would be required to update COMMENTs, which is wasteful.

Update the six `COMMENT ON COLUMN` strings listed above. These are documentation-only changes that don't affect behavior. If H1 is also fixed (Tier 2 fields removed from PATCH schema), the `daily_refresh_*` and `sse_*` comments should consistently say "Tier 2 — DB-only escape hatch."

### L2 — `invalidate()` schedules background refresh immediately, creating brief null-window

**File:** `apps/api/src/services/appConfig/cache.ts:164–171`

**What was found:**

`invalidate()` sets `_state.entry = null` then calls `void refresh().catch(...)`. Between the null-set and the refresh completing (one DB round-trip, ~1–5ms), any concurrent resolver call gets `null` from `getAppConfigCacheEntry()` and falls back to env. This is documented in the JSDoc comment ("acceptable per scope-todo"), but the concern is:

- The refresh promise is fire-and-forget (`void`), so if it fails, the next `getAppConfigCacheEntry()` call will schedule another refresh (because `_state.entry` is still null and `_state.loadedAt` is 0).
- This could cause a burst of DB reads if the PATCH handler is called rapidly under a DB outage.

**Recommendation (no immediate fix required):** The scope-todo accepts this behavior. Worth a comment noting the burst risk or adding a rate-limit on the background refresh trigger. Not blocking for this PR.
