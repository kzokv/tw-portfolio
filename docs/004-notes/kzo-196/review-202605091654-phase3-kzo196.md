## Findings

### HIGH-1 — `htmlFor`/`id` mismatch on sector dropdown

**File:** `apps/web/features/settings/components/InstrumentCatalogSheet.tsx`
**Lines:** 312, 318

The `<label>` references `catalog-sector-select` but the `<select>` element has
`id="catalog-sector-filter"`. The two strings do not match, so clicking the label
does not focus the select (broken accessibility, WCAG 1.3.1 / 4.1.2 Label).

```tsx
// line 312 — WRONG
<label htmlFor="catalog-sector-select" ...>

// line 318 — the actual id
<select id="catalog-sector-filter" ...>
```

**Fix:** Change line 312 to `htmlFor="catalog-sector-filter"` to match the `id` and the
`data-testid`.

### LOW-1 — `"asx-gics-csv" as never` cast in `asxGicsSyncWorker.ts`

**File:** `apps/api/src/services/market-data/asxGicsSyncWorker.ts`
**Line:** 73

```ts
await providerHealth.recordOutcome("asx-gics-csv" as never, outcome);
```

The `as never` cast suppresses a TypeScript error, likely because `ProviderHealthService.recordOutcome`'s
first argument is a discriminated union of known provider IDs and `"asx-gics-csv"` is not
yet in that union type.

The `as never` cast is not wrong at runtime (the implementation accepts any string), but it
defeats the type-checker for future callers and silences the signal that the union type needs
updating.

**Fix:** Add `"asx-gics-csv"` to the provider-ID union type that `recordOutcome` accepts,
then remove the cast. This is a 1–2 line change in the provider-health types.

**Alternatively:** accept the cast with an inline comment explaining why it is intentional
(e.g., "pending provider-health union type widening") — lower-risk option if the union type
is large or shared with the generated admin schema.
