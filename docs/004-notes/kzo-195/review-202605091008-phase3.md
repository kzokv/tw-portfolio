#### C1 — Missing `GET /admin/instruments` listing API endpoint

**File:** `apps/api/src/routes/adminRoutes.ts` + `apps/web/app/admin/instruments/page.tsx:12`
**Rule:** `interface-caller-verification.md` + `admin-new-subpage-checklist.md`

The web server component `page.tsx` fetches `/admin/instruments?marketCode=AU&page=1&limit=50` on every page load:

```ts
// apps/web/app/admin/instruments/page.tsx:12
const data = await getJson<AdminInstrumentsResponse>("/admin/instruments?marketCode=AU&page=1&limit=50");
```

Only two POST routes were implemented in `adminRoutes.ts`:
- `POST /admin/instruments/:ticker/:marketCode/undelete`
- `POST /admin/instruments/:ticker/:marketCode/exclude`

There is no `GET /admin/instruments` handler anywhere in the codebase (`grep -r 'instruments' apps/api/src/routes/adminRoutes.ts` shows no GET route with that pattern). The page will throw a 404 on every server-side render, rendering the entire `/admin/instruments` UI inaccessible.

**Required fix:** Implement `GET /admin/instruments` in `adminRoutes.ts` returning `AdminInstrumentsResponse` (paged list of `AdminInstrumentRow` from `persistence.listAdminInstruments()`). Also requires `listAdminInstruments()` in both `PostgresPersistence` and `MemoryPersistence` (persistence interface gap).

#### H2 — Admin session cookie discarded in E2E spec; browser remains in member role

**File:** `apps/web/tests/e2e/specs-oauth/admin-instruments-aaa.spec.ts`
**Rule:** `playwright-oauth-cookie-patterns.md` + `e2e-oauth-seed-as-browser.md`

`mintAdminCookie()` is called but its return value is never used:

```ts
// Approximately line 89 in admin-instruments-aaa.spec.ts
await mintAdminCookie();  // return value DISCARDED
await page.goto("/admin/instruments");
```

`mintAdminCookie()` presumably returns a cookie string or sets state that must be explicitly installed on `page.context()` via `page.context().addCookies([...])`. Without this, the browser carries the `oauthBase` fixture's default member session cookie. The admin instruments page applies `requireAdminRole(req)` — member access will be rejected (403 or redirect to `/`), so every assertion in the spec fires against a non-admin redirect page.

The correct pattern (from `playwright-oauth-cookie-patterns.md` + project precedent in sharing specs):

```ts
const { cookieHeader, cookieValue } = await mintAdminCookie();
await page.context().addCookies([{
  name: WebEnv.SESSION_COOKIE_NAME,
  value: cookieValue,
  domain: TestEnv.host,
  path: "/",
}]);
await page.goto("/admin/instruments");
```

**Required fix:** Capture return value from `mintAdminCookie()` and install the cookie on `page.context()` before navigating.

### LOW

#### L1 — `absent` metric includes excluded instruments; `absentTickers` excludes them — semantic inconsistency in logs and admin notifications

**File:** `apps/api/src/services/market-data/runCatalogSync.ts` + `apps/api/src/persistence/postgres.ts`
**Rule:** Informational (no named rule)

`CatalogSyncResult.absent` counts ALL instruments missing from the catalog (including `delisting_detection_excluded = TRUE`), while `CatalogSyncResult.absentTickers` contains only the candidate list (excluded rows filtered out). The admin notification message may show `absent=12, processed=7` with no explanation of the 5-row gap, which can confuse operators investigating anomalies.

**Suggested fix (non-blocking):** Either rename the metric to `absentCandidates` / `absentTotal` to make the distinction explicit in logs, or add a note to the notification message that excluded instruments are not counted in the processed figure. Can be addressed as a follow-up in Wave 2 docs / runbook addition.

#### L2 — Persistence-layer paranoia `throw new Error(...)` produces 500 in race condition

**File:** `apps/api/src/persistence/postgres.ts` (inside `undeleteInstrument` and `setInstrumentDelistingDetectionExcluded`)
**Rule:** `service-error-pattern.md`

The route correctly pre-guards with `routeError(404, "instrument_not_found", ...)` before calling persistence. However, the persistence methods contain an inner paranoia guard that throws a plain `new Error("instrument_not_found: ...")` for the TOCTOU case where the row disappears between the route's pre-check and the `UPDATE`. A plain `Error` has no `.statusCode`, so Fastify's error handler produces a 500.

This is a low-probability TOCTOU window (requires concurrent hard-delete between the route's read and the persistence write), but the rule is clear: persistence layer should use `routeError` or re-throw a structured error if it needs to express a non-500 status.

**Suggested fix:** Replace `throw new Error("instrument_not_found: ...")` in persistence with `throw routeError(404, "instrument_not_found", "Instrument not found")` imported from `apps/api/src/lib/routeError.ts`.
