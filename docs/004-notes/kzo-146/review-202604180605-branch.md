## Findings

### HIGH

#### H1. Server-side logout does not clear `tw_context_user_id`

**Location:** `apps/api/src/routes/registerRoutes.ts:1160-1167`

The Fastify `GET /auth/logout` route only clears the session cookie:

```ts
app.get("/auth/logout", async (req, reply) => {
  const attrs = buildCookieAttrs(Env.SESSION_COOKIE_NAME, ...);
  reply.header("set-cookie", `${Env.SESSION_COOKIE_NAME}=; ${attrs}; Max-Age=0`);
  ...
});
```

Design slice 17 requires:

> Clear `tw_context_user_id` cookie alongside session teardown. For server-side logout redirect path, include a `Set-Cookie: tw_context_user_id=; Max-Age=0` on the Fastify logout response.

Currently the cookie only clears because `UserAvatarButton.tsx:162` attaches an `onClick={() => clearContextCookie()}` handler. That covers happy-path UI logout but not:
- expired-session auto-logout via `redirectToLogoutOn401` (which *does* call `clearContextCookie` — OK)
- server-initiated force-logout (session_version bump)
- direct-URL navigation to `/auth/logout`
- any future logout code path that doesn't traverse the avatar button

**Fix:** In the `/auth/logout` handler, append a second Set-Cookie header for `tw_context_user_id`. `reply.header("set-cookie", ...)` with an array value propagates both cookies; or use the same inline-append pattern the `onSend` hook already uses at `app.ts:209-216`.

**Why this is High, not Medium:** Spec #13 (`switcher-logout-clears-cookie-aaa.spec.ts`) passes but only exercises the UI click path. A direct `window.location = "/auth/logout"` or server-redirect path leaks the context cookie across logout, meaning the next login briefly sees `x-context-user-id` for the previous user's selected share before the cookie eventually re-clears via the fallback path. Low-likelihood exposure, but trivially fixable and design-specified.

#### M2. SSE route cannot stamp `x-context-fallback` header

**Location:** `apps/api/src/routes/sseRoute.ts:110-116`, `apps/api/src/app.ts:206-217`

The `onSend` hook stamps the fallback header via `reply.header()`. That call queues the header in Fastify's internal buffer, to be flushed on `reply.send()`. The SSE route deliberately uses `reply.raw.writeHead(200, {...})` and bypasses `reply.send()`. Any fallback flagged on an SSE connect request is silently dropped.

In practice this is benign: `resolveSSEUserId` already uses `requireSessionUserId`, so a revoked `x-context-user-id` header doesn't affect the subscription target. The lazy fallback on subsequent `fetch()` calls catches the teardown. But the guarantee at `.claude/rules/fastify-raw-streaming-cors.md` — "any Fastify route that bypasses `reply.send()` must manually propagate CORS headers" — applies here too by analogy.

**Fix (choose one):**
1. Cheapest: add a comment in `sseRoute.ts` acknowledging that the fallback header is not propagated for SSE, and that enforcement relies on `requireSessionUserId` + lazy fallback from non-SSE fetches.
2. Proper: extend `pickCorsHeaders()` to also propagate `CONTEXT_FALLBACK_HEADER` and any `Set-Cookie` entries — keeping the `raw.writeHead()` call the sole place the head is flushed.

**Why Medium:** Masks a safety net. Fix is one grep away from becoming a regression next time someone adds a streaming route.

#### M3. `AppShell.tsx` complexity has grown materially

`apps/web/components/layout/AppShell.tsx` grew from ~920 to 1132 LOC with 6 new useEffects covering switcher state, fallback events, inbound refresh, deep-link handling, SSE revoke, and context-dependent refresh orchestration. Control flow for the switcher is spread across:

- `inboundShares` / `switcherLoaded` state (lines 101-102)
- `currentSharedOwner` / `isSharedContext` derivation (lines 119-143)
- `refreshSwitcherData` / `refreshContextDependentData` callbacks (lines 157-185)
- 4 useEffects (lines 187-243)
- `handleSharingNotification` / `handleContextSelect` callbacks (lines 351-377, 489-500)

**Fix:** Extract a `useSharedPortfolioContext()` hook (owns cookie, inbound shares, fallback listener, deep-link handler, refresh callbacks) and/or a `<PortfolioSwitcherController>` component that wraps the subtree needing switcher state. Keeps AppShell focused on layout.

**Why Medium:** Not a bug, but raises the cognitive load for the next KZO-148 (admin impersonation) change. AppShell is already the central orchestration surface — additions should land via focused hooks, not inline state.

#### M4. Switched-in empty-copy replacement leaks into metric-card detail slots

**Location:** `apps/web/components/layout/AppShell.tsx:130-143`, `791-815`, `955`

The `uiDict` shadow copies `dashboardHome.holdingsEmpty` with owner-facing copy ("{owner} hasn't added any holdings yet"). But that same dict key is reused for metric-card *detail* strings in non-empty paths:

- `largestPosition.detail` (line 808, 815)
- `concentration.detail` (line 955)

When a shared owner has holdings and a largest position, those detail slots still render owner-facing empty copy if the metric's primary source resolves to `null`. Most likely produces subtle mislabeling rather than correctness issues, because metric cards usually have a non-null value. Visible mostly in partial-data edge cases.

**Fix:** Introduce a dedicated `sharedHoldingsEmpty` key and use it only in the `holdingsEmpty` literal (owner-empty-state), keeping the existing `holdingsEmpty` for fallback detail slots. Alternatively, leave `holdingsEmpty` alone and switch the *empty-state* rendering in `HoldingsTable` to read from `dict.switcher.sharedHoldingsEmpty` directly.

**Why Medium:** Copy regression on shared read surfaces. Might not ship a visible bug but is the kind of thing that surprises QA.

#### M5. `PortfolioSwitcher` unit tests use a fragile testid selector

**Location:** `apps/web/test/components/layout/PortfolioSwitcher.test.tsx:149-153`

```ts
const ownerNodes = document.querySelectorAll("[data-testid^='portfolio-switcher-option-owner-']");
```

This matches the testid prefix `portfolio-switcher-option-owner-`. The component emits `portfolio-switcher-option-${item.ownerUserId}`. The assertion only works because fixture ownerUserIds start with `"owner-"` (e.g. `"owner-newer"`). Real production userIds are UUIDs — a future rename or realistic fixture will silently break the count assertion while still passing legacy lookups.

**Fix:** Assert on the specific testids (`portfolio-switcher-option-owner-newer`, `portfolio-switcher-option-owner-older`), or match on a non-overlapping sub-selector such as `[role="menuitem"]:not([data-testid='portfolio-switcher-option-self']):not([data-testid='portfolio-switcher-manage-sharing'])`.

**Why Medium:** Test brittleness, not a production bug. Cheap to fix now.

#### L3. `useNotifications.refetch` only surfaces the latest sharing notification as a callback

**Location:** `apps/web/hooks/useNotifications.ts:46-59`

`latestSharingNotification` is picked via `list.find(source === "sharing")` — server returns desc by `createdAt`, so this is the newest. Only that one ID is tracked in `latestSharingNotificationIdRef`, so two rapid sharing events (e.g. two revokes in close succession) will collapse into a single callback fire. Probably acceptable because the client only reacts to *revocation of currently-selected owner*, and `refreshContextDependentData` does a full refetch, but worth noting if we start relying on multi-event delivery.

#### L4. AppShell inbound-refresh effect piggybacks on `clearContextCookie()`'s internal event

**Location:** `apps/web/components/layout/AppShell.tsx:187-204`

The effect that clears the cookie when an inbound share vanishes does not dispatch `CONTEXT_CHANGED_EVENT` explicitly — it relies on `clearContextCookie()` dispatching the event internally. Correct as written but tightly coupled: removing the internal dispatch would silently break this effect. Worth a one-line comment noting the invariant.

**Location:** `apps/api/test/http/specs/switcher-narrow-taxonomy-aaa.http.spec.ts`

Scope-todo line 186-187 lists the assertions as:

**Fix:** Either extend the spec to assert `/notifications` and `/shares` under shared context, or file a follow-up ticket and update the scope-todo checkbox commentary.
