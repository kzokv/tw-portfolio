## MEDIUM Findings

### MEDIUM-1 — `retry-after` header uses `Env.*` constant instead of live resolver

**Rule:** `fastify-eviction-lifecycle-pattern.md §"Sweep parameter is admin-tunable; sweep cadence stays env-default"` — "the sweep parameter must be the live effective value, not the env-default captured at registration."

**Location:** `apps/api/src/routes/registerRoutes.ts` line 2573

```ts
// ❌ Uses env constant — not the live effective value
reply.header(
  "retry-after",
  String(Math.ceil(Env.ANONYMOUS_SHARE_RATE_LIMIT_WINDOW_MS / 1000)),
);
```

**Problem:** `assertAnonymousShareRateLimit(req.ip)` (in `anonymousShareRateLimit.ts`) correctly uses the live resolver `getEffectiveAnonymousShareRateLimitWindowMs()` for the window check. But the `retry-after` header in the catch block at line 2573 reads the env constant directly. When an admin overrides `anonymousShareRateLimitWindowMs` to, say, 120 000 ms (2 min), the check enforces 120 s but the `Retry-After` header tells clients to wait only 300 s (env default), or vice versa.

The `Retry-After` header is a runtime-response parameter (it communicates live policy to the client) — not a schedule cadence — so the rule's "parameter live" branch applies.

**Fix:**
```ts
// ✅ Live resolver
import { getEffectiveAnonymousShareRateLimitWindowMs } from "../services/appConfig/sharing.js";

reply.header(
  "retry-after",
  String(Math.ceil(getEffectiveAnonymousShareRateLimitWindowMs() / 1000)),
);
```

## LOW Findings

### LOW-1 — Misleading `useEffect` deps comment in `AdminSettingsClient.tsx`

**Location:** `apps/web/components/admin/AdminSettingsClient.tsx` lines 99–101

```ts
// `activeTab` intentionally omitted from deps: re-running on activeTab
// change would create a feedback loop with `setActiveTab` above.
}, [searchParams, activeTab]);  // ← activeTab IS in deps
```

**Problem:** The comment says `activeTab` is "intentionally omitted" but it is in the deps array. The code is functionally correct — the `if (fromUrl !== activeTab)` guard prevents the feedback loop the comment warns about — but the comment contradicts the implementation and will mislead future maintainers. The guard is the right solution; the comment should say "activeTab IS included to sync when it drifts from URL; the `!== activeTab` guard prevents re-triggering `setActiveTab` in a loop."

**Fix:** Update the comment to accurately describe the implementation:

```ts
// activeTab included in deps so the URL→state sync fires on both searchParams
// changes AND on back/forward navigation that changes activeTab without a URL
// update. The `fromUrl !== activeTab` guard prevents calling setActiveTab on
// every tab click (which would feedback-loop through the effect).
}, [searchParams, activeTab]);
```
