### Medium

**MEDIUM-1** `apps/api/test/integration/snapshots-multi-currency.integration.test.ts:203`

The backfill correctness test for the provisional row (`snap-3-prov`) uses JavaScript numeric coercion to assert the post-migration value:

```ts
// Comment on line 200 says: "value_native uses COALESCE(market_value, 0) per migration UPDATE script"
expect(Number(byId["snap-3-prov"].value_native)).toBe(0);
```

**Bug:** `Number(null) === 0` in JavaScript. When the pg driver returns a SQL NULL as JS `null`, `Number(null)` is `0` — the assertion passes trivially for both a true `0` value AND a SQL NULL. The test cannot distinguish between the two.

**Reality check:** The migration correctly uses `value_native = COALESCE(value_native, market_value)` where both are NULL for provisional rows → `value_native` is set to NULL, not 0. This is semantically correct per D9 (nullable column). But if the migration were later changed to incorrectly set `0` instead of NULL, this test would still pass.

The test comment also incorrectly references `COALESCE(market_value, 0)` from the scope-todo SQL rather than the actual migration SQL (`COALESCE(value_native, market_value)`).

**Recommendation:** Replace the assertion and fix the comment:

```ts
// snap-3-prov: provisional row. market_value is NULL, value_native starts NULL
// (just added column). COALESCE(value_native, market_value) = COALESCE(NULL, NULL) = NULL.
// value_native must be NULL on provisional rows per D10 — not 0.
expect(byId["snap-3-prov"].value_native).toBeNull();
expect(Number(byId["snap-3-prov"].cost_basis_native)).toBe(1000);
expect(byId["snap-3-prov"].unrealized_pnl_native).toBeNull();
expect(byId["snap-3-prov"].provider_source).toBe("finmind");
```

### Low

**LOW-1** `apps/api/test/unit/currencyWalletSnapshotGeneration.test.ts:50-58`

`getAllWalletSnapshotsForUser` accesses the `private readonly currencyWalletSnapshots` field of `MemoryPersistence` via a `as unknown as { currencyWalletSnapshots? }` type cast:

```ts
const memoryAny = persistence as unknown as {
  currencyWalletSnapshots?: CurrencyWalletSnapshot[];
};
const rows = memoryAny.currencyWalletSnapshots ?? [];
```

This is fragile: if the field is renamed in `memory.ts`, this silently returns `[]` (no TypeScript error, no test failure — just wrong results).

**Recommendation (preferred):** Add a `_getAllCurrencyWalletSnapshotsForUser(userId: string): CurrencyWalletSnapshot[]` synchronous helper to `MemoryPersistence` (mirroring the `_seedCurrencyWalletSnapshots` / `_clearCurrencyWalletSnapshots` helpers already present), then replace the unsafe cast with that helper in the test.

**Alternative (if the above is out of scope for a fix):** Build the read-back from the public API using the fixed account IDs already known to the test:

```ts
const acc1Rows = await persistence.getCurrencyWalletSnapshotsForAccount("user-1", "acc-1", "0000-01-01", "9999-12-31");
const acc2Rows = await persistence.getCurrencyWalletSnapshotsForAccount("user-1", "acc-2", "0000-01-01", "9999-12-31");
const rows = [...acc1Rows, ...acc2Rows].sort(...);
```

### Informational

**INFO-1** Migration backfill SQL deviates from scope-todo (improvement, not bug).

Scope-todo Phase 1 step 2 specified `value_native = COALESCE(market_value, 0)`. Migration uses `value_native = COALESCE(value_native, market_value)`. The migration's approach is more semantically correct: provisional rows (market_value IS NULL) get `value_native = NULL` instead of `0`. NULL accurately conveys "no close price available" while `0` would conflate "missing data" with "worth zero." No action needed; noting for KZO-166/176 reviewers who read the migration history.

**INFO-2** Dual-write equality test (`snapshotGeneration.test.ts:~466`) asserts `expect(s.valueNative).toBe(s.marketValue)`.

The implementation sets `marketValue = roundToDecimal(valueNative, 2)`. For integer close prices × integer quantities (all test data), `round(x, 4) === round(x, 2)` as JS numbers. The assertion is valid for all TWSE tick-size-constrained inputs (max 1 decimal place in price × integer quantity). The assertion encodes a slightly stronger invariant than D6 strictly requires (D6: legacy columns carry native values; not: they are numerically equal to the 4-decimal native column). No change required; worth noting for KZO-176 reviewers.

**INFO-3** `bulkUpsertHoldingSnapshots` uses `$15::timestamptz[]` UNNEST type for the `generated_at` column despite `daily_holding_snapshots.generated_at` being `TIMESTAMP` (without timezone). Postgres accepts the implicit cast. `bulkUpsertCurrencyWalletSnapshots` uses `$9::timestamp[]` which matches the DDL precisely. The inconsistency is inherited from pre-KZO-165 code. No action needed in this ticket.
