### Informational

**INFO-1** — Iter1 MEDIUM-1 was a false positive (documented for audit trail)

The iter1 reviewer stated "The migration correctly uses `COALESCE(value_native, market_value)`." The actual migration SQL is `COALESCE(market_value, 0)` — matching the scope-todo contract and making `value_native = 0` (not NULL) for provisional pre-migration rows. The test assertions at lines 204-205 are correct. No action.

**INFO-2** — `currency` column NOT NULL not added in migration

The migration promotes `daily_holding_snapshots.currency` to CHAR(3) + ISO CHECK but does not add a NOT NULL constraint. The column was `TEXT DEFAULT 'TWD'` and is now `CHAR(3)` (nullable). The ISO CHECK allows NULL through (Postgres treats `NULL ~ pattern` as UNKNOWN, not FALSE). The walker always emits a non-null `currency` from `trades[0].priceCurrency`, so in practice the column is always populated. The nullable semantics are intentional per D2 ("nullable for pre-v1 backfilled rows"). Noting for KZO-176 reviewers: when the column shape is finalized, adding NOT NULL then would be a trivial add-constraint migration.

**INFO-3** — `generateCurrencyWalletSnapshots` runs on both full-generation and scoped-recompute paths

`registerRoutes.ts:3348` (full snapshot-generate) and `registerRoutes.ts:3458` (recompute-confirm) both invoke the full wallet regeneration. The scoped-recompute path fires after a trade mutation, which does change cash ledger entries (trade settlement entries are written as part of the mutation). So the wallet re-aggregation on the recompute path is correct — it picks up the newly-written cash entries. The full-replace strategy is slightly more expensive than a scoped re-aggregate, but correct. KZO-166 should evaluate scoped wallet updates when it owns the WAC math.

**INFO-4** — Dual-write unit test (`snapshotGeneration.test.ts:459`) uses non-TWSE-constrained prices

The test asserts `expect(s.valueNative).toBe(s.marketValue)` using `closePrice: 100.1234`. For MemoryPersistence this holds because values are stored as plain JS numbers (no precision truncation). Against Postgres, `market_value NUMERIC(12, 2)` would store `300.37` while `value_native NUMERIC(20, 4)` would store `300.3702` — the equality would fail. The integration tests do not assert this equality against Postgres, so the integration suite does not cover this. Acceptable for the current ticket (D6 requires same-value at write time, not read-time equality), but KZO-176 reviewers should note this when reading the legacy column drop.
