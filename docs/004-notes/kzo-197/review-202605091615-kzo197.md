## Summary

KZO-197 wires the `yahoo-finance-au` "Re-run now" button to dispatch a **union** of (a) catalog warm-up over `instruments WHERE market_code='AU' AND bars_backfill_status IN ('pending','failed') AND delisted_at IS NULL` and (b) the existing monitored-AU refresh. The ticket also adds:
- `'awaiting'` as a 4th DTO-only status for fresh-deploy rows (both run timestamps null)
- Per-provider rerun cooldown: AU defaults to 30 min; others stay at 60 s
- `rerunCooldownMs: number` required field on the providers DTO
- Per-provider tooltip (`TooltipInfo`) with `{cooldown}` placeholder
- New `enqueueAuCatalogBarsBackfill` helper + `formatCooldownLabel` utility

## Findings

### MEDIUM

#### MEDIUM-1 — Integration tests use `buildApp({memory})` instead of scope-required `describePostgres`

**Files:**
- `apps/api/test/integration/auCatalogRerunUnion.integration.test.ts`
- `apps/api/test/integration/adminProvidersAwaiting.integration.test.ts`
- `apps/api/test/integration/perProviderRerunCooldown.integration.test.ts`

**Finding:**  
All three integration tests use `buildApp({ persistenceBackend: "memory" })` + `app.inject()`, which is the route HTTP pattern from `admin-management.integration.test.ts`. However, the locked scope-todo (`scope-todo-202605091500-locked.md` §4, Step 7) explicitly required:

> "Write a `describePostgres`-guarded integration test for `auCatalogRerunUnion` [...] Per `integration-test-persistence-direct.md`: these tests require Postgres to verify SQL-level filter correctness."

The test file headers cite `integration-test-persistence-direct.md` ("route HTTP tests do NOT require Postgres") but this selectively omits the same rule's broader context. The primary value of the Postgres-backed path for this ticket is to exercise `listAuCatalogBarsBackfillCandidates()` Postgres implementation — the SQL filter on `market_data.instruments WHERE market_code='AU' AND bars_backfill_status IN ('pending','failed') AND delisted_at IS NULL` with a real schema. The memory-backend implementation could silently diverge from the Postgres implementation without any test catching it.

**Impact:** `PostgresPersistence.listAuCatalogBarsBackfillCandidates()` SQL correctness is untested. Future regressions to the SQL predicate (e.g., wrong schema prefix, wrong status values) won't be caught by `npm run test:integration:full:host`. The existing `auCatalogSyncTwelveData.integration.test.ts` and `asxGicsCatalogSync.integration.test.ts` in the same worktree demonstrate the correct pattern for Postgres-backed catalog tests.

**Routing:** Phase 4 (Backend Implementer). At minimum, add a single `describePostgres` test verifying that `listAuCatalogBarsBackfillCandidates()` returns AU rows with `pending`/`failed` status and excludes delisted rows and non-AU rows using raw `pool.query` seeds per the established pattern in `integration-test-persistence-direct.md § Schema-qualified table names`.

## Additional Notes

**`enqueueAuCatalogBarsBackfill` implementation is clean.** The helper correctly:
- Reads candidates from `listAuCatalogBarsBackfillCandidates()` 
- Short-circuits when candidates list is empty (no-op, no `createRefreshBatch` call)
- Short-circuits when `boss === null` (memory backend) returning `{tickerCount:0, batchId:null}`
- Omits `startDate` per scope (worker resolves `historyStartFor('AU') = 1988-01-28`)
- Uses `${ticker}:${marketCode}` composite singleton key
- Allocates a refresh batch via `persistence.createRefreshBatch` when boss is present

**`formatCooldownLabel` is correct.** Boundary at 120,000 ms (120 s): ≤120,000 → `"${round(ms/1000)}s"`, >120,000 → `"${round(ms/60000)} min"`. Unit tests cover 0, negative, boundary values, and rounding. AU (1,800,000 ms) produces `"30 min"`; global default (60,000 ms) produces `"60s"`.

**`ProviderHealthStatusDto.status` union widening** (`"awaiting"`) correctly constrained to "route-derived only": `getProviderHealthStatus()` and `upsertProviderHealthStatus()` persistence types remain unchanged, as the `awaiting` value is only set in the GET route's DTO projection. The `provider_health_status` table `CHECK (status IN ('healthy','degraded','down'))` constraint is not affected.
