### HIGH

**H1: AccountCreateForm name input inside outer settings `<form>` — Enter key triggers settings save, not account creation**

- **File:** `apps/web/components/settings/SettingsDrawer.tsx:225–307`
- **What:** The accounts tab is rendered inside the outer `<form onSubmit={handleSubmit}>` block (the block that serves `general` and `fees` tabs via form-save). When a user types a name in `<input type="text">` inside `AccountCreateForm` and presses **Enter**, the browser fires the outer form's `onSubmit`, which calls `form.handleSubmit()` (the settings-save path), not the account-creation path. `AccountCreateForm`'s submit button correctly uses `type="button"`, but the name text input has no `onKeyDown` guard. If `form.isDirty` is true (user changed other settings before switching to Accounts tab), Enter would silently save those changes instead of creating the account — a confusing mutation.
- **Recommended fix (option A — minimal):** Add `onKeyDown={(e) => { if (e.key === "Enter") e.preventDefault(); }}` to the name text input in `AccountCreateForm.tsx:181`.
- **Recommended fix (option B — clean):** Move the `form.tab === "accounts"` conditional outside the outer form block (mirror the pattern of `profile`, `tickers`, `display` tabs). This also eliminates the `UnsavedChangesFooter` rendering on the accounts tab (which shows a Save button for general settings even while the user is creating an account).
- **Rule reference:** No specific rule, but is an established web form accessibility and UX invariant.

### MEDIUM

**M1: `UnsavedChangesFooter` rendered on accounts tab — Save button can save general settings while user is mid-account-creation**

- **File:** `apps/web/components/settings/SettingsDrawer.tsx:290–306`
- **What:** The `UnsavedChangesFooter` (with its Save/Discard/Saving… controls) is rendered for all tabs in the `(general || fees || accounts)` form block. If a user has unsaved fee-profile changes and switches to the Accounts tab, they see a "Save" button that saves the _settings_, not their account-creation form. The accounts tab form (`AccountCreateForm`) has its own submit button. Two "save" actions visible simultaneously is confusing.
- **Fix:** Same as H1 option B — move accounts tab outside the outer form block to get its own independent rendering path without `UnsavedChangesFooter`.
- **Severity rationale:** This is UX confusion, not data loss. Downcounted from HIGH because the settings Save button is labeled "Save" (general intent) not "Add account", reducing user confusion somewhat.

**M2: `accountsListSectionTitle` i18n value retains Fees-tab-scoped "Account Fallback Profile" label after move to Accounts tab**

- **File:** `apps/web/features/settings/i18n.ts:51` (en), `:206` (zh-TW)
- **What:** The i18n _key_ was correctly renamed from `accountFallbackSectionTitle` → `accountsListSectionTitle` per D11. However, the English _value_ remains `"Account Fallback Profile"` (zh-TW: `"帳戶預設費率"`). In the new Accounts tab context, this section represents "your accounts" generically, but the label still reads as a Fees-tab concept. Users landing on the new Accounts tab will see the form (for creation) and then a section labeled "Account Fallback Profile" — the "fallback" framing was Fees-tab-scoped per D11's rationale.
- **Suggested values:** `"Your Accounts"` / `"你的帳戶"` or `"Account Settings"` / `"帳戶設定"`. Final wording is implementer judgment.
- **Severity rationale:** Cosmetic/copy issue. No functional impact. Flagged MEDIUM because the scope-todo D11 explicitly cited "the 'fallback' framing was Fees-tab-scoped" as the rename motivation, making the retained value directly inconsistent with the documented intent.

- **What:** The `<section data-testid="account-create-form">` has no `aria-label` or `aria-labelledby`. The `<h3>` inside provides a visual label but there is no `aria-labelledby` linking the section to it. Screen readers announce it as a generic "section". LOW accessibility gap.

**L3: `handleSubmit` inside `AccountCreateForm` calls `onAccountsRefresh()` synchronously before `resetForm()` — potential ordering concern**

- **File:** `apps/web/features/settings/components/AccountCreateForm.tsx:125–127`
- **What:** `onAccountsRefresh()` is called before `resetForm()`. In practice, `onAccountsRefresh` triggers a background refetch (async), so `resetForm()` completes before the refetch result lands. React 18 batches the resulting state updates safely. This is fine and matches the D12 spec (`await onCreate(input); onAccountsRefresh(); resetForm()`). ✅ Confirmed no issue; documented for audit trail.
