## Deliverable 1 — Cash ledger rules reference doc

Create `docs/001-architecture/cash-ledger-rules.md` (~100-150 lines):

### Entry type table

- [ ] Document all 6 entry types in a table: type, enforced sign, generated by, required linkage
  - `TRADE_SETTLEMENT_IN` — `amount > 0`, generated by SELL trade booking + `replayPositionHistory`, requires `relatedTradeEventId`
  - `TRADE_SETTLEMENT_OUT` — `amount < 0`, generated by BUY trade booking + `replayPositionHistory`, requires `relatedTradeEventId`
  - `DIVIDEND_RECEIPT` — `amount > 0`, generated by `postDividend` / `updatePostedCashDividend`, requires `relatedDividendLedgerEntryId`
  - `DIVIDEND_DEDUCTION` — `amount < 0`, generated by `postDividend` / `updatePostedCashDividend`, requires `relatedDividendLedgerEntryId`
  - `MANUAL_ADJUSTMENT` — any sign, reserved (no active code path), neither FK required
  - `REVERSAL` — any sign, reserved (no active code path), requires `reversalOfCashLedgerEntryId`
- [ ] Note that `DEPOSIT` and `WITHDRAWAL` are planned as future first-class types (not yet in schema) per scope-grill Q6 decision

### Sign conventions

- [ ] Document the DB CHECK constraints that enforce sign per entry type (reference `baseline_current_schema.sql:93-157`)
- [ ] Document the `CHECK (amount <> 0)` constraint — zero-amount entries are invalid (cf. replay invariant 4)
- [ ] Document `NUMERIC(20, 2)` precision — supports fractional ETF settlement amounts
- [ ] Note: **MemoryPersistence has no sign enforcement.** Sign conventions are DB-enforced only. Dev/test mode (`PERSISTENCE_BACKEND=memory`) silently accepts sign violations. The spec should flag this as a known limitation.

### Entry generation matrix

- [ ] Trade settlement generation paths:
  - `createTransaction` in `portfolio.ts` → `buildTradeSettlementCashEntry` → one entry per trade
  - `replayPositionHistory` → reconstructs settlement entries with `cash-replay-{tradeId}` id pattern
  - `confirmRecompute` in `recompute.ts` → `replaceCashLedgerEntryForTrade` when fees change
- [ ] Dividend cash generation paths:
  - `postDividend` in `dividends.ts` → `buildDividendCashLedgerEntries` → one RECEIPT + N DEDUCTIONs
  - `preparePostedCashDividendUpdate` → same builder, entries replaced via delete-and-replace
  - Id pattern: `{dividendLedgerEntryId}:receipt` and `{dividendLedgerEntryId}:deduction:{deductionId}`
- [ ] Document deduction subtypes that produce DIVIDEND_DEDUCTION entries: NHI_SUPPLEMENTAL_PREMIUM, WITHHOLDING_TAX, BROKER_FEE, BANK_FEE, TRANSFER_FEE, CASH_IN_LIEU_ADJUSTMENT, ROUNDING_ADJUSTMENT, OTHER
- [ ] Note: **zero-amount deductions must be filtered before creating cash entries.** `buildDividendCashLedgerEntries` creates `amount: -deduction.amount` for each deduction. If `deduction.amount = 0`, this produces a zero-amount cash entry violating `CHECK (amount <> 0)`. The client-side form filters zero-amount deductions before submission. The `dividend_deduction_entries` table has `CHECK (amount > 0)` which catches it in Postgres. MemoryPersistence has neither guard. (Code fix is a separate bug — spec documents the requirement.)
- [ ] Note entry ID collision prevention: trade cash entries use `source="trade_settlement"`, dividend cash entries use `source="dividend_posting"`. The unique index `ux_cash_ledger_entries_account_source_reference` on `(account_id, source, source_reference)` separates the namespaces.

### Linkage rules

- [ ] Trade entries: `relatedTradeEventId` required, `relatedDividendLedgerEntryId` must be NULL (enforced by DB CHECK)
- [ ] Dividend entries: `relatedDividendLedgerEntryId` required, `relatedTradeEventId` must be NULL (enforced by DB CHECK)
- [ ] MANUAL_ADJUSTMENT / REVERSAL: neither FK required
- [ ] Document `ON DELETE CASCADE` from `trade_events` → `cash_ledger_entries` (hard delete removes linked cash entries)
- [ ] Document that dividend cash entries are NOT cascaded — they're managed by the dividend posting service
- [ ] Document `receivedCashAmount` derivation: not stored on `dividend_ledger_entries` (column removed in migration 010). Computed on every read as `COALESCE(SUM(amount) FROM cash_ledger_entries WHERE entry_type = 'DIVIDEND_RECEIPT' AND related_dividend_ledger_entry_id = dle.id, 0)`. This means deleting or editing DIVIDEND_RECEIPT cash entries retroactively changes the reported receivedCashAmount.

### Correction model

- [ ] **Cash ledger entries are mutable derived state.** The source of truth is the originating event (trade event or dividend ledger entry), not the cash entry itself.
- [ ] Entries can be edited and deleted. The reversal pattern defined in the canonical accounting model (line 391) is overridden for Phase 1 — entries are directly replaced rather than reversed.
- [ ] Document the sanctioned patterns:
  - Trade settlements: `replayPositionHistory` deletes all TRADE_SETTLEMENT_IN/OUT for (account, ticker) and rebuilds from the current trade set. This is a full reconstruction, not a point correction.
  - Dividend cash entries: `updatePostedCashDividend` in persistence deletes all cash entries linked to the dividend ledger entry and re-inserts from the updated posting. Delete-and-replace is the sanctioned pattern.
- [ ] REVERSAL type remains in schema but has no active code path. Documented as reserved for potential future use.

### Replay reconstruction semantics

- [ ] Document how `replayPositionHistory` interacts with cash entries:
  - Step 4 in replay: `deleteTradeCashEntriesForAccountTicker` removes all TRADE_SETTLEMENT_IN/OUT for the scoped (account, ticker)
  - Step 5: replay loop generates new settlement entries per trade
  - Step 6: `bulkInsertCashLedgerEntries` persists the rebuilt set (with invariant 4: filter zero-amount entries)
  - Dividend cash entries are NOT touched by replay — they're managed by the dividend posting service independently
