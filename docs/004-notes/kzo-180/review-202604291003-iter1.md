# Review: KZO-180 Reporting Currency

## Verdict

No blocking correctness findings after takeover review.

## Scope Reviewed

- `user_preferences.preferences.reportingCurrency` schema and resolver path.
- `/dashboard/overview` summary translation and `/dashboard/performance` FX-aware time series.
- Postgres and memory persistence aggregation semantics, including D8 self-pair handling.
- Shared dashboard DTO changes and frontend dashboard label consumers.
- Display-tab reporting-currency selector and refresh wiring.
- KZO-180 unit, integration, HTTP, web unit, and OAuth E2E coverage.
- KZO-167/KZO-179 forward-note cleanup and evergreen architecture docs.

## Review Notes

- `totalCostCurrency` is no longer present in wire DTO callers; remaining hits are comments or negative-shape assertions.
- `fxAvailable=false` performance points now gate every wire numeric to `null`, preventing Postgres partial-SUM leaks when one contributor is self-pair and another lacks FX.
- The two HTTP route-level missing/self-pair cases are intentionally skipped because the OAuth HTTP suite cannot seed non-TW instruments today; the same contract is covered by unit tests and Postgres integration tests.
- `code-reviewer` tooling was run. `pr_analyzer.py --base dev --head HEAD --json` reported `no_changes` because the KZO-180 changes are still uncommitted working-tree changes. `code_quality_checker.py apps/api/src/services --language typescript --json` flagged high complexity in the new reporting-currency service; this is a maintainability note, not a correctness blocker for this handoff. The service is isolated by design for KZO-176 absorption and has direct focused tests.
