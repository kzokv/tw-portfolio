## Findings

### MEDIUM-1 — Persistence-DTO partial-sum divergence between Memory and Postgres

`apps/api/src/persistence/postgres.ts:2869-2898` and `apps/api/src/persistence/memory.ts:1769-1801` produce different intermediate values when `fxAvailable === false` AND some contributors had FX:

- **Memory**: `costSum`, `cumRealSum`, `cumDivSum` are NOT accumulated — final values are `0`.
- **Postgres**: SQL `SUM(value * CASE...)` ignores NULL multiplications, so the persistence DTO carries a **partial sum** from rows whose FX did resolve.

Both are masked at the wire layer (`p.fxAvailable ? p.totalCostBasis : null` in `dashboardReportingCurrency.ts:309-315`), so the API contract is consistent end-to-end. INT-3 documents the divergence in a comment (`apps/api/test/integration/dashboardReportingCurrencyAggregation.integration.test.ts:357-371`).

**Risk**: any future direct consumer of `getAggregatedSnapshotsInReportingCurrency()` that bypasses the wire layer (e.g. an internal report, KZO-176 dashboard rewrite) could read inconsistent values across backends.

**Recommendation (deferred)**: align the Postgres mapper to coerce `totalCostBasis`, `cumulativeRealizedPnl`, `cumulativeDividends` to `0` when `fx_available=false`, mirroring the memory impl. One-line change at `postgres.ts:2890-2894`. Not blocking — the test suite documents the divergence and the wire layer hides it. Track for KZO-176 absorption.

### MEDIUM-2 — Route handler reads `getUserPreferences` twice on `/dashboard/performance`

`apps/api/src/routes/registerRoutes.ts:3270-3273`:

```ts
const [{ ranges }, prefs] = await Promise.all([
  resolveEffectiveRanges(app.persistence, userId),  // calls getUserPreferences internally
  app.persistence.getUserPreferences(userId),       // duplicate read
]);
```

`resolveEffectiveRanges` already calls `persistence.getUserPreferences(userId)` at `userPreferences.ts:53`. So every `/dashboard/performance` request issues two identical reads against the user_preferences row.

**Impact**: low — Postgres row is single-key indexed; ~1ms each. But `resolveEffectiveRanges` is the hot path for the timeframe picker.

**Recommendation**: read prefs once at the top, pass into a refactored `resolveEffectiveRanges(persistence, userId, prefs?)`. Defer if not blocking — it's a 4-line refactor that touches one signature. Suggested for a follow-up cleanup ticket or absorption into KZO-176.

### LOW-1 — `/dashboard/overview` serial prefs read

`apps/api/src/routes/registerRoutes.ts:3252` reads `prefs` strictly after `loadUserStore` + `buildDashboardOverview` complete. Could parallelize with `resolveQuoteSnapshots` (line 3241) — 1 round-trip saved per request. Same priority as MEDIUM-2.

### LOW-2 — Dead conditional branch in `translateOverviewSummary`

`apps/api/src/services/dashboardReportingCurrency.ts:154` initializes `let totalCostAmount: number | null = 0` but is never reassigned to `null` in the function body. The `(totalCostAmount ?? 0)` on line 217 is therefore equivalent to `totalCostAmount`. Type-only artifact; not buggy. Could simplify to `let totalCostAmount = 0` and drop the `?? 0`.

Same observation applies to `marketValueAmount`, `unrealizedPnlAmount`, `dailyChangeAmount`, `upcomingDividendAmount` — all start at `0`, are guarded by `Has*Missing` flags, and the `?? 0` coalescing is unreachable. Minor maintainability.

### LOW-3 — Synthetic fallback hardcodes cumulative fields to 0

`apps/api/src/services/dashboardReportingCurrency.ts:506-507`:

```ts
cumulativeRealizedPnlAmount: allFxAvailable ? 0 : null,
cumulativeDividendsAmount: allFxAvailable ? 0 : null,
```

This is correct (no snapshot data → 0 cumulative). The hardcode is intentional but worth a one-line comment confirming "no snapshot data path → cumulative fields are 0 by definition" so a future maintainer doesn't read it as a TODO.

### LOW-4 — `fxStatus` rollup at `apps/api/src/services/dashboardReportingCurrency.ts:95`

`if (fxMap.size === 0) return "complete";` — empty contributors return "complete." The HTTP-1 spec asserts this exact contract, but the in-code comment doesn't justify why "no contributors → complete" was chosen over "no contributors → missing." Add a one-line comment: "empty portfolio is unambiguously complete (no FX needed)." Defensive against future debate.

### LOW-5 — DisplayTabSection `useEffect` empty-dep hydration race

`apps/web/components/settings/DisplayTabSection.tsx:71-87` hydrates from GET on mount with `[]` deps. If the parent re-mounts the component (e.g. SettingsDrawer re-renders on locale change), the GET fires again. The component currently uses `cancelled` flag to handle unmount-mid-flight, but a fresh GET could race a recent PATCH from another tab/window. Not a bug; just an observation. Same pattern exists for SortableCardGrid per the comment on line 64. Acceptable.

## Residual Risk

Carried from iter 1, still applicable:

- **Overview `totalCostAmount` is non-null by DTO contract** even when `fxStatus === "partial"`. The wire layer surfaces a partial sum (in reporting currency) without a null signal. The UI is expected to read `fxStatus` to decide whether to label "(partial)" — but no current UI surface does this. KZO-176 should decide whether to expose `fxStatus` visually.
- **HTTP-5 + HTTP-8 are skipped** because the OAuth HTTP test catalog only seeds TW-market instruments (`memory.ts:185-191`) and `/__e2e/seed-instruments` is gated to `dev_bypass` mode. The same contract is regression-guarded by INT-3 (missing FX) + INT-1 (self-pair) at the persistence layer + DRC unit tests at the service layer. Promotion path documented in the spec.

New residual:

- **Persistence partial-sum divergence (MEDIUM-1)** — if KZO-176 (or any other consumer) exposes the persistence DTO directly, it must explicitly gate on `fxAvailable` before reading `totalCostBasis`/`cumulativeRealizedPnl`/`cumulativeDividends`. The wire layer in `dashboardReportingCurrency.ts` does this correctly today; downstream consumers must follow suit.
- **Performance**: route handlers double-read `user_preferences` on `/performance` and serially read on `/overview`. Negligible at current scale; revisit if `/dashboard/*` p99 grows.
