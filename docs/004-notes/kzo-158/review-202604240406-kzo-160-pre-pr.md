## H1 — Ticker hygiene violation in new E2E specs (HIGH)

**Files:**
- `apps/web/tests/e2e/specs/transaction-price-prefill-aaa.spec.ts:9,10,13,34-38`
- `apps/web/tests/e2e/specs/transaction-fee-estimate-aaa.spec.ts:10-15, 51-55`

The three new AAA specs seed daily bars for reserved tickers:
- `transaction-price-prefill-aaa.spec.ts` test 1: seeds `2330` at `2026-01-15`
- `transaction-price-prefill-aaa.spec.ts` test 2: seeds `2317` at `2026-01-16`
- `transaction-fee-estimate-aaa.spec.ts` test 1: seeds `0050` at `2026-01-15`
- `transaction-fee-estimate-aaa.spec.ts` test 2: seeds `2330` at `2026-01-15`

The project rule `.claude/rules/e2e-shared-memory-bars-ticker-hygiene.md` explicitly reserves `2330`, `2317`, and `0050` — and the same PR already refactored `portfolio-snapshots-aaa.spec.ts` to use synthetic tickers `8201`–`8206` for exactly this reason (new top-of-file comment: "Use a unique synthetic ticker per charted case so snapshot walks do not inherit bars seeded by earlier tests"). The new specs regress that.

`MemoryPersistence` keeps daily bars in a process-global array; `getLatestBars(ticker, n)` spans every `_seedDailyBars` call in the process. Concrete risks:
- `dashboard-daily-change-aaa.spec.ts` expects `/No market data/i` for `2330` (line 25). If the price-prefill spec runs first, the `2026-01-15` bar lingers and that assertion fails.
- `portfolio-transactions-aaa.spec.ts` seeds `2330` and `2317` catalog entries. Any cross-pollination of bar data hits the same flake pattern the rule was written to stop.

**Fix:** swap to unused synthetic tickers (e.g. `8301`, `8302`, `8303`, `8304`) after grep-verifying the fresh choices:

```bash
grep -rn '"8301"' apps/web/tests/e2e/ apps/api/test/
```

Update the seed-instruments name to match ("8301" etc., use any plausible name). This is a ~10-line change across the two spec files.

## H2 — Out-of-scope change to `ActionsMixin` default navigation (HIGH)

**File:** `libs/test-framework/src/mixins/ActionsMixin.ts:15-30`

The PR silently changes `waitUntil` from `"domcontentloaded"` to `"commit"` on three shared helpers used by every E2E spec in the monorepo:

- `mxGotoUrl` — no follow-up app-ready wait
- `mxNavigateToRoute` — followed by `mxWaitForAppReady()`
- `mxReloadPage` — followed by `mxWaitForAppReady()`

`"commit"` resolves as soon as the navigation response is received (pre-parse, pre-DOMContentLoaded). Two concerns:

1. **Scope creep.** The scope-todo has no item authorizing a test-framework-wide timing change. This is a blast-radius move landing inside a UI polish ticket.
2. **Silent regressions.** `mxGotoUrl` has no follow-up readiness wait. Any spec that calls `mxGotoUrl` and immediately interacts with the page now races DOM parse. This likely manifests as intermittent "locator not found" flakes rather than hard failures, which is the worst class of regression.

The same hunk also breaks indentation — the three `@Step` decorators and bodies sit at 2-space indent while the surrounding class members are at 4-space. Either reflow the block or revert.

**Fix:** revert the three `waitUntil` changes. If the change is actually load-bearing for KZO-160 (nothing in the scope-todo suggests this), extract it to a dedicated ticket with explicit E2E regression checking.

## M2 — Prewarm route additions are partially invalid (MEDIUM)

**Files:**
- `libs/test-e2e/src/fixtures/base.ts:16-19`
- `libs/test-e2e/src/fixtures/publicPages.ts:14-17`

Two concerns:

1. **`publicPages.ts` prewarms `/share/prewarm-token`.** There is no real share token with that name; this path will 404 on every prewarm call. Prewarm will compile the route template (useful), but leaving a literal placeholder in checked-in fixture code is confusing and invites the "why does this 404 appear in every run?" debug loop. Use a documented sentinel (e.g. `/share/__PREWARM__`) or an actual-shaped-but-clearly-synthetic token, and add a comment explaining the intent.

2. **`base.ts` adds four `/admin/*` prewarm calls to the shared dev_bypass base.** The base fixture is shared by non-admin tests too. Admin routes require admin role and redirect otherwise. As a prewarm compile trigger this works (Next.js compiles the page regardless), but (a) it couples every spec to admin route compilation and (b) the `admin-portal-aaa.spec.ts` changes in this PR also add `test.slow()` calls that look like they're papering over the same slowness this prewarm is meant to fix — both shouldn't be needed. Pick one or explain both.

**Scope concern:** neither prewarm change is called out in the KZO-160 scope-todo. Consider extracting or justifying.

## M4 — Rename flow wipes in-progress settings edits (MEDIUM)

**Files:**
- `apps/web/features/settings/components/AccountFallbackSection.tsx:60-71` (`saveRename` calls `onRenameAccount`)
- `apps/web/components/layout/AppShell.tsx:486-489` (`handleRenameAccount` calls `dashboard.refresh()`)
- `apps/web/features/settings/hooks/useSettingsForm.ts:50-66`

Renaming an account triggers `dashboard.refresh()`. On refresh completion, `useSettingsForm`'s effect fires (deps include `accounts`, `feeProfileBindings`, `feeProfiles`) and calls `setDraft(toSettingsFormModel(...))` — blowing away any unsaved fee-profile dropdown changes, binding edits, or other draft state the user had made in the drawer.

The newly-added `wasOpenRef` correctly keeps the user on the "fees" tab, but the draft reset is a real regression of the in-drawer workflow (example: user picks a fee-profile fallback for account A, then renames account B → their unsaved A choice vanishes).

This may be pre-existing behavior exposed by the new rename path; either way the rename path is the new trigger and should not compound it.

**Fix options:**
- Merge rename result into `draft.accounts` directly instead of rebuilding `draft` from the refreshed snapshot, OR
- Don't call `dashboard.refresh()` from the rename handler — let the settings PATCH's response feed the local state and let the next explicit save / drawer-reopen pick up the authoritative snapshot.

## L4 — `match: "previous"` for FinMind exact-date hits is semantically odd (LOW / INFORMATIONAL)

**File:** `apps/api/src/routes/registerRoutes.ts:1170-1178` (`buildFetchedPriceLookupResponse`)

Per the scope-todo (§F2 step 4), FinMind-fallback responses always carry `match: "previous"` and a `reason`, even when the returned bar's date exactly matches the requested date. The integration test "keeps provider fallback responses in previous-match mode even on exact-date hits" explicitly pins this behavior. Scope-authorized, but worth flagging for future-reader clarity:

- A consumer of the API looking at `{ date: "2026-01-16", match: "previous", reason: "no_bar" }` has to know that "previous" here means "our DB didn't have it, the provider did" — not "the date doesn't match."

Consider either renaming (`source_tier: "db" | "provider"`) or at minimum adding a one-line JSDoc on the response type documenting the scope decision. Low effort, high future-clarity.
