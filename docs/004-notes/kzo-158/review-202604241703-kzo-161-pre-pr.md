## HIGH (blocks PR)

### HIGH-1 ‚Äî Unit tests are non-operative (SortableCardGrid.test.tsx, every it-block)
**File:** `apps/web/test/components/layout/SortableCardGrid.test.tsx` ‚Äî lines 242-261, 353-384, 447-476  
**Issue:** All three core test blocks (debounce coalesce, rollback on PATCH failure,
last-server-confirmed baseline) guard their assertions behind two conditions that are
never true:
1. `container.querySelector("[data-testid='sortable-card-grid']")` ‚Äî the `SortableCardGrid`
   component renders no element with this testid.
2. `typeof element._testOnDragEnd === "function"` ‚Äî the component exports no escape hatch.

Every assertion falls through to:
```ts
console.info("[SortableCardGrid.test] No _testOnDragEnd escape hatch found ‚Äî skipping ‚Ä¶");
```
Tests pass vacuously. The design doc ¬ß5 explicitly states "Unit test target covers optimistic
rollback injection" and ¬ß16 lists this as a coverage contract.

**Recommendation (owner: implementer):**  
Add either (a) a `data-testid="sortable-card-grid"` wrapper div on the root `<div>` inside
`DndContext` and a `_testOnDragEnd` DOM property that calls an exposed setter, or (b) extract
the debounce + rollback machinery into a `useSortableOrder` hook and unit-test the hook
directly with `renderHook`. Option (b) is cleaner and avoids DOM-property hacks.
The tests as written already express the correct contract ‚Äî only the integration wiring
is missing.

### HIGH-2 ‚Äî `card-reorder-aaa.spec.ts`: state read-back uses wrong user's session cookie
**File:** `apps/web/tests/e2e/specs-oauth/card-reorder-aaa.spec.ts` ‚Äî [card-A] line 166-173, [card-B] line 232-237  
**Issue:** `getCardOrder(session.cookieHeader)` reads `/user-preferences` using the fresh session
minted for `"card-reorder-A-sub"` / `"card-reorder-B-sub"`. But the browser PATCH is made as
`testUser` (the fixture's user). The GET returns preferences for `card-reorder-A-sub`, which has
no `cardOrder` entry ‚Äî so `order` is always `null`. The `expect.poll` will time out for every
run of [card-A] and [card-B].

Concrete flow:
1. `mintSessionCookie({ sub: "card-reorder-A-sub" })` ‚Üí `session` is `card-reorder-A-sub`
2. `seedUserPreferences(session.cookieHeader, testUser.userId, {})` ‚Üí seeds for `testUser.userId` ‚úì
3. Browser drag PATCH ‚Üí updates `testUser.userId`'s `cardOrder` ‚úì
4. `getCardOrder(session.cookieHeader)` ‚Üí reads `card-reorder-A-sub`'s preferences ‚Üí `null` ‚úó

**Recommendation (owner: qa):**  
To get a cookie for `testUser.userId`, either:
- Use `page.context().cookies()` to extract the session cookie already in the browser jar and
  pass it to `getCardOrder`, or
- Extract the testUser's sub/email from the fixture and mint a cookie for them specifically
  (note: `testUser.userId` alone is not enough ‚Äî `GET /user-preferences` requires a session
  cookie, not a header).

The `dashboard-timeframe-aaa.spec.ts` [timeframe-L] had the same obstacle (see MEDIUM-1
below) and fell back to DOM-only assertions. The card-reorder spec needs the same fallback
or a proper cookie-extraction strategy.

## MEDIUM (should fix before PR)

### MEDIUM-1 ‚Äî `[timeframe-L]` state read-back is dead code
**File:** `apps/web/tests/e2e/specs-oauth/dashboard-timeframe-aaa.spec.ts` ‚Äî lines 238-246  
**Issue:** `prefsResponse` is fetched using `headers: { "x-e2e-user-id": testUser.userId }`.
This header is not a valid auth mechanism in OAuth mode (`specs-oauth/` runs `AUTH_MODE=oauth`).
`resolveUserId` will reject the unauthenticated request or ignore the header. The result is
never asserted on ‚Äî the spec falls back to DOM-only pill visibility checks (lines 249-253),
which are weaker than the design's "state read-back" requirement.

**Recommendation (owner: qa):**  
Delete the dead `prefsResponse` block. Decide: (a) keep DOM assertions only and add a comment
explaining the cookie-extraction obstacle, or (b) implement the same fix as HIGH-2 to get
`testUser`'s actual session cookie and assert `preferences.dashboardPerformanceRanges`.

### MEDIUM-2 ‚Äî `timeframe-chip-remove-{range}` testid not in scope-todo testid registry
**File:** `apps/web/components/admin/AdminSettingsClient.tsx` ‚Äî line 383  
```ts
toggleTestId={(r) => `timeframe-chip-remove-${r}`}
```
**Issue:** The admin's `SortableRangeList` emits a second per-row button with testid
`timeframe-chip-remove-{range}`. This is a new testid that does not appear in the scope-todo
testid registry (design ¬ß11 "Added" table). The admin AAA spec does not reference it
(the existing `clickAdminTimeframeChip(range)` action still clicks the chip button, which
is the primary admin toggle affordance). The `timeframe-chip-remove-{range}` button is a
redundant second affordance with no test coverage.

**Recommendation (owner: implementer or qa):**  
Either: (a) add `timeframe-chip-remove-{range}` to the testid registry and the admin spec, or
(b) remove the `toggleTestId` prop from the admin's `SortableRangeList` consumer ‚Äî admin
already has the chip button as its toggle, so the second button is unnecessary and creates
untested UI. Option (b) also reduces DOM surface.

### MEDIUM-3 ‚Äî `SortableCardGrid.test.tsx` second test block lacks `card-drag-handle-*` filter in `getCardOrder`
**File:** `apps/web/test/components/layout/SortableCardGrid.test.tsx` ‚Äî lines 441-444  
**Issue:** The `getCardOrder` helper in the second `describe` block ("uses last-server-confirmed
state") queries `[data-testid^='card-']` without filtering out `card-drag-handle-*` or
`card-inner-*` testids:
```ts
return Array.from(cards).map((el) => el.getAttribute("data-testid")?.replace("card-", "") ?? "");
```
This differs from the first `describe`'s `getCardOrder` (lines 338-348) which does filter.
If the escape-hatch fix (HIGH-1) is applied, this test will include drag-handle and inner-card
elements in the returned order array, causing assertion mismatches. The implementer's own
memory entry (`kzo-161-card-testid-namespace-collision`) documents this exact pattern ‚Äî it
was applied inconsistently.

**Recommendation (owner: qa):**  
Apply the same filter logic from the first test block to the second test block's `getCardOrder`.

## LOW (nice to fix)

### LOW-1 ‚Äî `useEffect` dependency array omits `cards` in `SortableCardGrid`
**File:** `apps/web/components/layout/SortableCardGrid.tsx` ‚Äî line 144  
```ts
}, [orderKey]); // `cards` intentionally omitted ‚Äî see inline comment
```
**Issue:** The initial-hydration `useEffect` uses `cards` (for `mergeCardOrder`) but lists
only `[orderKey]`. The comment explains the intent (re-hydrating on every `cards` identity
change would defeat the optimistic display order). A second `useEffect` on `[cards]`
compensates by re-aligning display order when the canonical list changes (lines 148-153).
This is architecturally sound but will produce an `eslint-plugin-react-hooks/exhaustive-deps`
warning that may fail the `npx eslint .` gate.

**Recommendation (owner: implementer):**  
Add `// eslint-disable-next-line react-hooks/exhaustive-deps` with a comment explaining why
`cards` is intentionally omitted, or restructure to avoid the exhaustive-deps false positive
(e.g. pass a stable `orderKey`-based key to the effect and use a ref for `cards`).
