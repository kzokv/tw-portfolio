# Todo: Frontend Redesign Reliability

> For agents starting a fresh session: this file is the locked scope. Where it conflicts with older dashboard/reporting notes, this file wins.

## Release Invariant

Every financial number rendered across Dashboard, Reports, Portfolio, Ticker, and non-admin user-facing/share pages must be backend-authoritative, FX-aware, market-aware, and honestly labeled when incomplete.

Do not silently fall back, synthesize trend data, or recompute financial totals in page-local frontend code. The frontend may format values and manage UI state, but financial totals, market values, P&L, allocation, daily change, return, cost basis, and reporting-currency conversions must come from trusted backend DTOs/read models.

Ticker correctness is not specific to `2330`. Use `2330` as a sanity check because it reproduced the issue, but all tickers and markets must render market value, unrealized P&L, account breakdown, and related reporting-currency values correctly after the fix.

Before final delivery, cross-check every item in this locked scope and the active gap note `gap-fix-20260611-dashboard-chart-grid-followups.md`. Any unchecked, unimplemented, or unevidenced item must be reported as a remaining gap rather than implied complete.

## Locked Scope

1. Ship one large PR covering data correctness, snapshot repair, shared holdings UI, global Quick Actions, Reports/Portfolio/Ticker fixes, i18n, and tests.
2. Trend charts are strict snapshot-only. Do not fall back to current holdings or synthetic trend points.
3. If expected snapshot/price data is unavailable, show a small tooltip/status using market-aware expected trading date when feasible; otherwise show the latest available snapshot date for the selected scope.
4. Fix the snapshot pipeline end to end: forward scheduling/triggering, user-owned regeneration, and admin/system bulk repair/backfill for affected users/date ranges.
5. Users can regenerate snapshots for their own current editable portfolio/context. Admin/system jobs handle broad repair/backfill across users.
6. Backend financial DTOs/read models are the calculation authority, with performance safeguards: no N+1 per-card/per-row calls, prefer shared aggregation paths, and surface incomplete state rather than blocking full pages.
7. Introduce a shared holdings grid foundation and migrate Dashboard Top Holdings, Portfolio Holdings, Reports Top Movers/Holdings Detail/Concentration/Top Holdings/Market Detail to it.
8. Persist holdings grid settings in backend user preferences, not localStorage. Settings include style preset, per-context column order, and per-context column widths.
9. Holdings grid settings are user-specific and keyed per table context, for example `dashboard.topHoldings`, `portfolio.holdings`, and `reports.holdings`.
10. Use style presets instead of two long-lived implementations: dashboard defaults to compact; portfolio/reports default to detailed; user override is allowed.
11. Column header drag reorders columns. Column resizing adjusts width. Text wraps automatically when width is narrow.
12. UI/UX must follow existing shadcn conventions and primitives where available: Button, Select, Tabs, Sheet, Popover, Tooltip, Badge, Card, Table, Dialog, Command, and related variants.
13. Desktop and mobile usability are both first-class acceptance criteria. Desktop favors dense scannable tables/toolbars; mobile favors stacked cards, bottom sheets, wrapped text, reachable controls, and no horizontal page overflow.
14. Promote Quick Actions globally across authenticated editable app pages and hide it entirely in shared/read-only contexts.
15. Move all current Dashboard quick actions global: Add transaction, Recompute portfolio, Generate snapshots, plus Change reporting currency.
16. Quick Actions is the only primary global reporting-currency switcher. Pages display active/resolved currency and may open Quick Actions, but must not host competing primary currency selectors.
17. Reports controls simplify to scope and range. Reports should show resolved reporting currency and direct users to Quick Actions to change global reporting currency.
18. Reports currency behavior: scope `all` uses global reporting currency; single-market scope uses that market's native currency.
19. Dashboard, Portfolio, Reports, and Ticker pages must display the active/resolved reporting currency near relevant cards/tables.
20. Ticker page scope is targeted: fix market value/unrealized P&L/account breakdown correctness for all tickers and redesign Account breakdown responsiveness. Do not do a full ticker page redesign.
21. i18n cleanup covers all non-admin product and public/share user-facing routes, excluding admin settings/admin tooling/dev-only pages.
22. Health column becomes clarified Data health: quote status, FX status, and snapshot/freshness reliability, with tooltip/detail explanation.
23. Data health is visible by default in detailed style. Compact-style handling may collapse or summarize it.
24. Worst P&L remains a sort mode, clarified as Worst P&L first, and keeps all rows visible.
25. Attention coloring is semantic only: positive/negative P&L and deltas, severity badges, and stale/missing quote/FX/snapshot states. Plain market value, cost basis, and allocation stay neutral unless status-bearing.
26. Allocation fallback must be explicit. If allocation falls back to cost basis, show the basis label and actual amount used. No silent fallback.
27. Reports keep specialized holdings cards and full Holdings detail cards, all migrated to the shared holdings grid with clear labels/presets.
28. Out of scope: investment advice, target allocations, buy/sell/hold recommendations, admin settings i18n, full ticker redesign, new chart types, cross-browser polish beyond validation, and bulk historical market data acquisition where provider data does not exist.
